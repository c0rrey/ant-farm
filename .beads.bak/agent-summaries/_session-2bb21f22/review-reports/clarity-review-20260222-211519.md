## Findings Catalog

### Finding 1

- **File:Line**: `orchestration/RULES.md:111`
- **Severity**: P3
- **Category**: Misleading key name (magic label)
- **Description**: The progress log key `tasks_approved=<N>` is a carry-over from the pre-change version of Step 1b, when user approval was explicit. After this commit, "approval" is automatic — the key name implies a human approved the task list, which is no longer accurate. A developer reading `progress.log` entries or parsing the log in `scripts/parse-progress-log.sh` would see `tasks_approved` and assume a user was involved.
- **Suggested fix**: Rename the key to `tasks_accepted=<N>` or `tasks_scheduled=<N>` to reflect that the count is the number of tasks the Queen will proceed with, not tasks a user explicitly approved.

### Finding 2

- **File:Line**: `orchestration/RULES.md:99-100`
- **Severity**: P3
- **Category**: Contradictory adjacent statements
- **Description**: Line 99 states "auto-approve regardless of task count" but line 100 immediately carves out task count = 0 as an exception. A reader following line 99 literally would auto-proceed on a zero-task briefing, then hit line 100 reversing that decision. The contradiction is on consecutive lines and requires a mental pause to reconcile. The zero-task guard is functionally unambiguous in isolation, but the preceding blanket "regardless of task count" is too broad given the exception that follows.
- **Suggested fix**: Qualify line 99 to exclude zero: "No complexity threshold applies; auto-approve regardless of task count (except zero — see Zero-task guard below)." Alternatively, move the zero-task guard sentence to precede "regardless of task count" so the exception is stated before the general rule.
