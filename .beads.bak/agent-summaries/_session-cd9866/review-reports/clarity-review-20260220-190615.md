## Findings Catalog

### Finding 1: `install-hooks.sh` blocks on missing scrub script but pre-commit hook silently skips it
- **File(s)**: `scripts/install-hooks.sh:74-77` vs `docs/installation-guide.md:46-47`
- **Severity**: P2
- **Category**: clarity
- **Description**: The behavior described in `docs/installation-guide.md:46-47` ("Skips silently if `scrub-pii.sh` is not found or not executable") contradicts what the installed pre-commit hook code actually does at `install-hooks.sh:74-77`: it exits with status 1 and a "Commit blocked." error message. A developer reading the docs would expect silent skip behavior; the actual code blocks the commit with an error. The discrepancy is a P2 because a developer encountering the blocked commit would be confused and might waste time troubleshooting.
- **Suggested fix**: Update `docs/installation-guide.md:47` to match the actual behavior: "Blocks the commit with an error if `scrub-pii.sh` is not found or not executable."
- **Cross-reference**: Correctness reviewer should also examine this — it is a functional mismatch, not merely a doc gap.

### Finding 2: `pantry.md` Section 2 deprecation notice lacks instructions for the reader's next action

- **Category**: clarity

A reviewer reading just RULES.md would calibrate their P1/P2 against web concepts.

### Finding 11: `install-hooks.sh` non-fatal sync failure has no rationale comment
- **File(s)**: `scripts/install-hooks.sh:44-46`
- **Severity**: P3
- **Category**: clarity
- **Description**: The pre-push hook silently treats `sync-to-claude.sh` failures as non-fatal — it prints a WARNING and continues the push (reported by excellence-reviewer). There is no comment explaining why this design choice was made. A developer reading lines 44-46 would reasonably wonder: "If the sync fails, shouldn't the push be blocked to keep `~/.claude/` in sync?" The intentional non-blocking decision is not self-evident.

- **Cross-reference**: Flagged by excellence-reviewer.

### Finding 13: `PLACEHOLDER_CONVENTIONS.md` audit table claims PASS for `RULES.md` but misses a Tier 3 convention violation

Update the audit table status from PASS to PASS-WITH-EXCEPTION and add a note column entry.

## Preliminary Groupings

### Group A: Stale reader/consumer annotations (root cause: documentation not updated when architecture changed)
- Finding 7 (`reviews.md:1`) — reader comment names a consumer that no longer exists
- Finding 8 (`README.md:171-176`) — architecture diagram shows Pantry in review mode (deprecated)
- Finding 2 (`pantry.md:271-277`) — Section 2 deprecation notice is incomplete

All three stem from the same underlying cause: when the `pantry-review` agent was deprecated and replaced by `fill-review-slots.sh`, the consumer-facing annotations and diagrams were not fully updated.

### Group B: Placeholder/term definition gaps (root cause: term definition blocks not maintained as new placeholders are introduced)
- Finding 5 (`big-head-skeleton.md:13`) — `{REVIEW_ROUND}` formatting inconsistency in term def block
- Finding 6 (`checkpoints.md:514`) — `{SESSION_START_DATE}` missing from term definitions block

Both stem from the same cause: term definition blocks are not treated as a living inventory that must be updated when new placeholders are added.

### Group C: Documentation vs. implementation mismatch (root cause: prose docs not updated when code behavior changed)
- Finding 1 (`install-hooks.sh:74-77` vs `docs/installation-guide.md:46-47`) — documented behavior ("skips silently") contradicts implemented behavior ("blocks commit")

### Group D: Context-specific prose in general-use files
- Finding 4 (`RULES.md:407-413`) — P2 severity examples use web-UI vocabulary in an orchestration-only file

### Group E: Structural organization and stale compliance records (root cause: living documents not updated when referenced files or commands change)

- Finding 13 (`PLACEHOLDER_CONVENTIONS.md:104` / `RULES.md:248-253`) — audit table claims PASS but misses a Tier 3 convention violation in the Hard Gates table prose

### Group F: Undocumented design intent / magic values (root cause: non-obvious decisions and values lack rationale or explanation)

- Finding 11 (`install-hooks.sh:44-46`) — non-fatal sync failure is an intentional design choice with no comment explaining why

Neither theme is blocking, but the stale compliance record is the more ironic gap given that document's stated purpose.
