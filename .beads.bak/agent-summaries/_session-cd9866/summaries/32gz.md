# Task Summary: ant-farm-32gz
**Task**: SESSION_ID collision: same-second Queens produce identical session directory
**Commit**: 28ea7e1

## 1. Approaches Considered

### Approach A — Add $RANDOM to entropy pool (SELECTED)
Change `echo "$$-$(date +%s%N)"` to `echo "$$-$(date +%s%N)-$RANDOM"`. `$RANDOM` is a POSIX bash/zsh built-in producing 0-32767, adding 15 bits of independent randomness.
- **Pro**: Minimal change — one token added to an existing expression
- **Pro**: `$RANDOM` is built into bash/zsh; no external command dependency
- **Pro**: Works even when `date +%s%N` degrades to literal 'N' on older macOS BSD date
- **Pro**: Same formula can be used in both RULES.md and PLACEHOLDER_CONVENTIONS.md
- **Con**: $RANDOM is not cryptographically random; collision probability ~1/32767 per same-PID same-second pair

### Approach B — Use uuidgen with Linux fallback
`SESSION_ID=$(uuidgen 2>/dev/null || cat /proc/sys/kernel/random/uuid 2>/dev/null | shasum | head -c 8)`. Uses OS UUID generation.
- **Pro**: UUID v4 has 122 bits of randomness — effectively zero collision probability
- **Con**: `uuidgen` is not universally available on all Linux distros; requires fallback chain
- **Con**: More complex one-liner with multiple subshells and fallback logic
- **Con**: Larger change; harder to audit at a glance

### Approach C — Use /dev/urandom
`SESSION_ID=$(head -c 8 /dev/urandom | shasum | head -c 8)`. Drops PID and timestamp entirely; uses kernel CSPRNG.
- **Pro**: Cryptographic quality randomness; effectively zero collision probability
- **Pro**: Available on both macOS and Linux
- **Con**: Drops PID and timestamp from the hash input, losing diagnostically useful context
- **Con**: Slightly larger change (replaces the entire expression)

### Approach D — Increase hash output length from 8 to 12 chars
Keep same generation formula but `head -c 12` instead of `head -c 8`. Increases hash space from ~4 billion to ~281 trillion.
- **Pro**: Trivial one-character change
- **Con**: Does NOT fix the root cause: if `%N` is literal 'N' and PIDs match, same input → same 12-char hash
- **Con**: Just reduces (but doesn't eliminate) collision probability; doesn't address the degenerate-input case

## 2. Selected Approach with Rationale

Selected **Approach A**. The bug's primary failure mode is same-PID + same-second with `%N` degrading to a literal. Adding `$RANDOM` breaks the input symmetry: even when two Queens share a PID and start in the same second with the same `%N` literal, their `$RANDOM` values will differ (with high probability). The change is the smallest possible modification to the existing expression, is cross-platform, and requires no new external dependencies. The 1/32767 same-RANDOM collision probability in the degenerate case is acceptable given the additional `$$` and `%s%N` entropy that already exists in non-degenerate conditions.

## 4. Correctness Review

### orchestration/RULES.md:L290

The new expression `echo "$$-$(date +%s%N)-$RANDOM" | shasum | head -c 8`:
- `$$`: Current process PID — same Queens in same PID namespace on same machine may collide, but two Claude Code sessions running simultaneously typically get different PIDs
- `$(date +%s%N)`: Epoch nanoseconds on Linux and modern macOS; epoch+literal-'N' on older BSD date — still provides second-level uniqueness
- `$RANDOM`: 0-32767, re-evaluated at expansion time — 15 bits of entropy independent of PID and timestamp
- `shasum | head -c 8`: SHA-1 hash, truncated to 8 hex chars — 32-bit hash space

**Worst-case collision probability** (same PID, same second, `%N` = literal 'N'): 1/32767 ≈ 0.003%. Acceptable for session directories.

**Acceptance criterion 1**: SESSION_ID generation includes enough entropy to avoid collisions with same-second launches — PASS. $RANDOM adds independent entropy that breaks input symmetry even in the degenerate same-PID + same-second case.

**Acceptance criterion 2**: Same-second Queens produce different session directories on both macOS and Linux — PASS. Verified empirically: rapid-fire generation produced unique 8-char IDs (7c19ed87, 2ef98479, 8ee06f6c, 33224d3b, 53bf0127). Collision simulation confirmed: old formula with same input produced identical hash; new formula with different $RANDOM values produced distinct hashes.
