## Findings Catalog

### R2-EC-01
**File**: `skills/plan.md:L140`
**Severity**: P2
**Original finding**: EC-05 (heredoc delimiter collision)
**Fix applied**: Replaced `cat > file <<'SPEC_EOF' ... SPEC_EOF` with `printf '%s\n' '<INPUT_TEXT>'`
**Assessment: FIX INCOMPLETE ‚Äî new injection vector introduced**

The heredoc delimiter collision is resolved, but `printf '%s\n' '<INPUT_TEXT>'` creates a new shell injection vulnerability. The `<INPUT_TEXT>` placeholder is wrapped in single quotes in the instruction prose. When an agent substitutes the actual spec content into this command, if the spec contains a single quote character, the shell command breaks with a syntax error (`unmatched '`), causing the input.txt write to fail silently or abort the session.

Example: a spec containing `"It's a requirement"` ‚Üí shell sees `printf '%s\n' 'It's a requirement'` ‚Üí syntax error.

The original EC-05 problem (heredoc delimiter collision) was an unlikely edge case. This replacement creates a more common failure: any spec with an apostrophe or single quote breaks the write. Real-world specs routinely contain contractions and possessives.

**Correct fix**: Use a shell variable, not a literal substitution:
```bash
printf '%s\n' "${INPUT_TEXT}" > "${DECOMPOSE_DIR}/input.txt"
```
This relies on the variable being set in the shell environment before the command runs, which is how the skill's flow works (INPUT_TEXT is captured earlier in the skill execution).

### R2-EC-02
**File**: `skills/plan.md:L127-L134`
**Severity**: P2
**Original finding**: EC-06 (JSON injection via unescaped INPUT_SOURCE)
**Fix applied**: Replaced heredoc JSON with `jq -n` construction
**Assessment: FIX CORRECT for string fields, but `--argjson class_score <CLASS_SCORE>` has a latent risk**

The `jq -n` approach correctly sanitizes string fields (`INPUT_SOURCE`, `INPUT_CLASS`, `DECOMPOSE_ID`). However, `--argjson class_score <CLASS_SCORE>` uses `--argjson` (raw JSON value) rather than `--argjson` with a shell-expanded integer. The `<CLASS_SCORE>` placeholder is described as a numeric score (0‚Äì6). At runtime, the agent substitutes this value directly. If the substitution produces a non-integer (e.g., due to a parsing error in Step 2), `jq` will reject the `--argjson` argument and exit non-zero, leaving `manifest.json` unwritten.

This is P2 because the class score is agent-computed (not user-supplied), so the range of values is narrow, but the failure mode (missing manifest.json, no error message surfaced to the user) is silent and could cause downstream confusion.

**Better fix**: Use `--arg class_score "<CLASS_SCORE>"` (string) and let consumers parse it as a number, or validate that `CLASS_SCORE` is an integer before the `jq` call.

### R2-EC-03
**File**: `scripts/setup.sh:L140-L157`
**Severity**: P3
**Original finding**: EC-02 (find exit code not handled in process substitution)
**Fix applied**: Replaced process substitution with `find > tmpfile`, checks exit code
**Assessment: FIX CORRECT, minor resource leak on tmpfile under specific failure path**

The fix correctly captures `find` exit code and surfaces a diagnostic. However, when `find` fails, the tmpfile is created by `mktemp` but may retain a partial (non-empty) error message from stderr captured via `2>&1`. The `rm -f "$find_output"` in the failure branch correctly cleans it up ‚Äî no leak in the failure branch.

One edge case: if the script is killed by a signal (SIGTERM, SIGKILL) between `mktemp` and the `rm -f` cleanup in either branch, the tmpfile (`/tmp/tmp.XXXXXX`) is left on disk. Under `set -euo pipefail`, unexpected failures do not run cleanup. This is P3 ‚Äî tmpfiles in `/tmp` are ephemeral and cleaned on reboot, and this is a common limitation of the shell pattern.

**Suggested improvement** (not blocking): Use a trap to clean up the tempfile:
```bash
find_output=$(mktemp)
trap 'rm -f "$find_output"' EXIT
```
But this is a polish item, not a runtime failure condition in normal usage.

### R2-EC-04
**File**: `scripts/setup.sh:L68`
**Severity**: P3 (previously untracked)
**Assessment: NEW FINDING introduced by fix**

The `backup_and_copy` function now uses `return 1` (fix for EC-03). Call sites in Step 3 (orchestration loop, L154) and Step 1 (agents loop, L113) now correctly handle the return value with `|| { warn ...; continue; }`. However, the `backup_and_copy` call at L177 (build-review-prompts.sh install) and L193/L210 (crumb.py and CLAUDE.md installs) does NOT have a `|| { warn ...; continue; }` guard. If backup fails for any of those files, `backup_and_copy` returns 1, and with `set -euo pipefail` active, the script aborts immediately without logging which file caused the failure.

This is P3 because backup failure on those files is unlikely and the abort message from `set -e` would reference the line number. But it is inconsistent with the handling added for the agent and orchestration loops.

**Suggested fix**: Add `|| warn "Failed to install: $BRP_DST"` (and similar) on the unguarded `backup_and_copy` calls.
