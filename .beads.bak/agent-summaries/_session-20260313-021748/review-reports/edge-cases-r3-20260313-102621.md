## Fix Verification

### R2-EC-01 fix: `printf '%s\n' "${INPUT_TEXT}"`

**Single quotes / apostrophes**: `printf '%s\n' "${INPUT_TEXT}"` with `INPUT_TEXT="it's a spec, don't skip"` outputs the string correctly. Shell expansion occurs before `printf` receives the argument — there is no shell parsing of single quotes inside the expanded value. Fix is correct.

**Double quotes, backticks, dollar signs**: All pass through correctly. `printf '%s\n'` uses the literal `%s` as format; `INPUT_TEXT` is passed as the argument, not as part of the format string. No format-string injection risk.

**Backslash sequences**: `printf '%s\n' "${INPUT_TEXT}"` with `INPUT_TEXT='path\nfile'` outputs `path\nfile` literally — the `%s` format does not interpret backslash escapes in the argument. This is the correct behavior for preserving spec content verbatim.

**Newlines in INPUT_TEXT**: Multi-line spec content (`$'line1\nline2\nline3'`) is preserved correctly. `printf '%s\n'` appends one trailing newline; internal newlines in the variable are output as-is.

**Empty INPUT_TEXT**: Produces a file containing a single newline (1 byte). This is benign — the pre-flight guard at `skills/plan.md:L81-L85` hard-stops on empty or whitespace-only `INPUT_TEXT` before Step 3 is reached. Empty `INPUT_TEXT` cannot arrive at the `printf` call under normal skill execution.

**Percent signs**: `INPUT_TEXT='100% done %s %d'` outputs correctly. The `%s` format specifier is fixed in the `printf` format string argument; `INPUT_TEXT` is the value argument, not the format string.

**Very large input**: No truncation — `printf` handles arbitrarily long strings within shell limits. No issue.

### R2-EC-02 fix: `--arg class_score "<CLASS_SCORE>"`

Changing from `--argjson` (raw JSON) to `--arg` (string) means `class_score` is now stored as a JSON string (`"4"`) rather than a JSON integer (`4`) in `manifest.json`. This is a type change. Any downstream consumer that reads `manifest.json` and expects `class_score` to be a JSON integer will now receive a string.

**Is there a downstream consumer?** Searching the skill: `manifest.json` is written in Step 3 and is not read again within `skills/plan.md`. It is a session artifact for audit purposes. No downstream agent or workflow step within the reviewed file set reads `manifest.json` programmatically. The `class_score` field is surfaced to the user as display output (Step 2: `score: CLASS_SCORE/6`), not used for logic. The type change from integer to string in the JSON file is safe for the current codebase.

**Is the substituted value still meaningful?** `<CLASS_SCORE>` is now quoted: `--arg class_score "<CLASS_SCORE>"`. The agent substitutes the integer value (0–6) as a string. `jq` accepts any string via `--arg` — no failure risk regardless of what the agent substitutes. Fix is strictly more robust than the previous `--argjson` approach.
