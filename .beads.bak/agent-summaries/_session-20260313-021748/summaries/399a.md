## 1. Approaches Considered

### Approach A: Thin Agent Definition + Monolithic Template
Single thick template in `orchestration/templates/surveyor.md` combining all
workflow content and prompt text. Agent definition references it. No separate
skeleton — the skeleton content would be inlined into the template.
- Pro: Single source of truth
- Con: Skeleton content would either duplicate the template or be absent; does
  not match the established pattern where skeletons are the spawn-ready artifacts

### Approach B: Layered — Agent Definition + Workflow Template + Prompt Skeleton (Selected)
Agent definition describes identity and points to template. Template is the
detailed workflow reference the agent reads mid-turn. Skeleton is the
Queen-spawnable prompt with `{UPPERCASE}` placeholders.
- Pro: Matches scout/pantry/dirt-pusher precedent exactly; clean separation of concerns
- Con: Three files to maintain; skeleton and template share some content (by design)

### Approach C: Agent Definition + Combined Template+Skeleton (Two-File Collapsed)
Two files: agent definition + one file serving as both workflow reference and
skeleton. Achieves both with fewer files.
- Pro: Less total content; fewer files
- Con: Skeleton placeholders (`{UPPERCASE}`) would pollute the workflow reference;
  the Queen-for-spawn vs. agent-read-for-workflow distinction breaks down

### Approach D: Example-Heavy Pattern Library
Build the workflow template as a pattern library rather than a procedural guide —
leading with extensive good/bad output examples throughout, with procedural steps
as secondary structure.
- Pro: Rich example coverage (AC#7); easy to scan for the right pattern
- Con: Bloats the workflow document; procedural clarity suffers; examples are
  better co-located with the step they apply to rather than in a separate library

---

## 2. Selected Approach

**Approach B: Layered (Agent Definition + Workflow Template + Prompt Skeleton)**

Rationale:
1. Exactly matches the established scout-organizer.md → scout.md and pantry-impl.md
   → pantry.md + dirt-pusher-skeleton.md architecture already in the repo.
2. Separates what the Queen sees (skeleton, filled before spawn) from what the
   agent reads mid-turn (template, read at execution time).
3. Brownfield handling, question prohibitions, and good/bad examples belong in the
   template (detailed reference), but their key constraints are also surfaced in
   the skeleton so the agent sees them immediately without reading the full template.

## 3. Implementation Description

Three files created:

### agents/surveyor.md
Agent definition with YAML frontmatter: `name: surveyor`, `model: opus`,
`tools: Read, Write, Glob, Grep, Bash, AskUserQuestion`. Body describes the agent
identity, the three input values (feature request, decompose dir, codebase root),
and a 5-step workflow overview pointing to the template for full details.

### orchestration/templates/surveyor.md
Full 6-step workflow template:
- **Step 1** (Brownfield): Scan `{CODEBASE_ROOT}` directory structure and top-level
  configs. Build internal context map. Explicit brownfield rule: codebase-answered
  questions must not be asked.
- **Step 2** (Parse): Extract explicit requirements, constraints, non-requirements,
  and genuine ambiguities from the feature request.
- **Step 3** (Q&A): Question prohibitions list (8 categories of banned questions),
  good-question checklist (4 criteria), 12-question limit with priority ordering,
  question format spec, and good/bad question examples.
- **Step 4** (Synthesize): Spec quality gates for requirements and acceptance
  criteria, invented-requirements prohibition, vague-criteria ban list.
- **Step 5** (Write spec.md): Exact output format with Scope/Constraints/Requirements
  (REQ-N with AC-N.M)/Non-Requirements/Assumptions/Open Questions sections.
  Good/bad examples for requirements, acceptance criteria, and non-requirements.
- **Step 6** (Return): Compact return summary format for the Queen.
- **Error handling**: Empty request, user skipping Q&A, unreadable codebase,
  missing decompose dir.

### orchestration/templates/surveyor-skeleton.md
Queen-spawnable prompt with:
- Instructions-for-Queen header (not sent to agent)
- Term definitions block per PLACEHOLDER_CONVENTIONS.md
- Placeholder list: `{DECOMPOSE_DIR}`, `{CODEBASE_ROOT}`, `{FEATURE_REQUEST}`
- Agent-facing section (below `---` separator) with feature request block,
  5-step workflow summary, critical constraints (all prohibitions), output format
  reference, good/bad examples (acceptance criteria, questions, requirements), and
  return format.
