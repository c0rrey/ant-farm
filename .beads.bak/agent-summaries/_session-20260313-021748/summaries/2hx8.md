## 1. Approaches Considered

### Approach A: Minimal trigger-and-delegate
A thin skill with a description field and a single instruction: "read `orchestration/RULES.md` and follow it." No inline logic.
- Pro: Zero maintenance burden; RULES.md changes propagate automatically.
- Con: Provides no guidance on pre-flight checks, error handling, or SESSION_DIR creation. The model must infer all startup behavior from RULES.md, which is written for the Beads workflow, not the crumbs workflow. High risk of the model skipping the coherence check or using wrong SESSION_DIR path.

### Approach B: Full inline workflow replication
Copy every step from RULES.md into the skill body with crumbs substitutions applied throughout.
- Pro: Fully self-contained, no cross-file dependency at runtime.
- Con: ~400 lines of redundant content. Instantly out of sync with RULES.md. Severe DRY violation. Not maintainable.

### Approach C: Structured dispatcher with coherence check inline + RULES.md delegation (selected)
Skill contains: pre-flight error handling (3 hard-stop checks), execution startup coherence check (3 warning-level checks), SESSION_DIR creation with correct `.crumbs/sessions/` path, and a delegation section pointing to RULES.md with crumbs-specific override notes.
- Pro: Right-sizes the skill — inline only what's crumbs-specific and stable, delegate the rest. Clear structure. All acceptance criteria satisfied. Coherence check is documented exactly where it belongs.
- Con: Requires the model to read two documents (skill + RULES.md) at runtime. Acceptable tradeoff given the skill explicitly calls this out.

### Approach D: Skill with inline bash execution for dynamic context
Use Claude Code's `!` bash syntax to run `crumb list --open` etc. at invocation time, injecting live task data before the model reads the workflow steps.
- Pro: Model sees actual live task data, fewer interpretation errors.
- Con: Bash execution in skills is atypical and fragile if `crumb` CLI isn't installed. The skill becomes a functional dependency on CLI state, not just a guidance document. Better suited for commands, not skills. Adds complexity without sufficient benefit since the Scout runs CLI commands anyway.

## 2. Selected Approach

**Approach C** — Structured dispatcher with coherence check inline + RULES.md delegation.

Rationale: Skills are guidance documents for the model. The pre-flight and coherence check logic is domain-specific to the crumbs migration and stable — it doesn't change when RULES.md evolves. Inlining it in the skill ensures it's always executed and clearly documented. The orchestration workflow is complex and maintained in RULES.md; duplicating it would create a maintenance hazard. The skill bridges the gap by providing crumbs-specific overrides (SESSION_DIR path, CLI commands) while delegating the 400+ lines of orchestration rules to their canonical source.
