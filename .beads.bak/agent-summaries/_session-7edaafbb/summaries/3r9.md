## 1. Approaches Considered

1. **Append PID (`$$`)** — `$(date +%Y%m%dT%H%M%S).$$` — the shell's own PID is unique among all concurrent processes on a POSIX system. If two sync runs happen in the same second, they will always have different PIDs. Portable across macOS BSD date and GNU date.

2. **Use nanoseconds (`%N`)** — `$(date +%Y%m%dT%H%M%S%N)` — eliminates collision risk with nanosecond precision, but `%N` is a GNU date extension not available on macOS without installing coreutils. Non-starter for a macOS-targeted repo.

3. **Use `mktemp`** — generate a temp file with mktemp and rename it to the backup path. Collision-resistant but requires different code structure (mktemp creates a file immediately, requiring a two-step create-then-move). Over-engineered for this use case.

4. **Append `$RANDOM`** — `$(date +%Y%m%dT%H%M%S).$RANDOM` — adds a random 0-32767 suffix. While collision probability is ~1/32768 per second, PID is preferred because it is deterministic and meaningful (identifies the writing process).

## 2. Selected Approach

**Option 1: Append PID.**

Rationale: PID is guaranteed unique among concurrent processes. Portable across macOS and Linux without any external dependencies. The suffix is meaningful — it identifies which process created the backup. Single-character change (append `.$$`) minimizes diff noise.
