# Task Summary: ant-farm-lc97

**Task**: fill-review-slots.sh resolve_arg accepts empty file content without warning
**Commit**: 781a592

## Approaches Considered

1. **Check file size with `-s` before cat (inside resolve_arg)** — After the file-exists check, add `[ ! -s "$fpath" ]` to detect 0-byte files. Simple and matches POSIX `-s` semantics (true if file exists and has size > 0). Error message includes the file path for debuggability. Chosen approach.

2. **Check output of cat after assignment in caller** — Don't change `resolve_arg`; instead check `CHANGED_FILES` and `TASK_IDS` after they are assigned. Separates detection from the reading logic, but requires the caller to know whether the value came from a file or a literal. Also, if a literal empty string were passed directly, this check would flag it incorrectly. Rejected.

3. **Capture cat output and test for empty string** — Inside `resolve_arg`, do `local content; content="$(cat "$fpath")"; if [ -z "$content" ]; then ...`. This catches whitespace-only files too, but adding string trimming (to detect all-whitespace) adds complexity beyond what the acceptance criteria require. `-s` is simpler and directly targets the stated 0-byte file bug.

4. **Return sentinel exit code from resolve_arg and have caller check** — Have `resolve_arg` exit non-zero for empty files and let the caller handle it. Adds error-propagation boilerplate; since the function already calls `exit 1` for missing files, using the same pattern for empty files is more consistent. Rejected in favor of consistency with existing error handling.

5. **Warn to stderr but continue** — Print a warning and let the script continue with empty content. The acceptance criteria says "warning or error". However, continuing silently with blank slot fills in review prompts is a worse outcome than a clear failure — the prompt would be malformed with no indication of why. Error (exit 1) is chosen over warning.

## Selected Approach

**Approach 1 — `-s` check inside `resolve_arg`, after file-exists check.**

Rationale: Co-located with the file reading logic, consistent with the existing missing-file error pattern in the same function, precise error message with file path, exits immediately before any processing of the empty value.

## Implementation Description

Added a four-line guard after the file-exists check in `resolve_arg` (L86-89):

```bash
if [ ! -s "$fpath" ]; then
    echo "ERROR: @file argument is empty (0 bytes): $fpath" >&2
    exit 1
fi
```

`-s` returns true if the file exists and has size greater than zero. This check fires only in the `@*` branch (file-reference arguments), so literal string arguments are unaffected. The check runs before `cat "$fpath"`, so no I/O occurs for empty files.
