# Task Summary: ant-farm-40z

**Task**: rsync --delete silently removes custom user files from ~/.claude/orchestration/
**Commit**: e445a40

## 1. Approaches Considered

1. **Remove --delete entirely** — simplest fix; rsync only adds/updates, never removes. User custom files are fully preserved. Stale source-tracked files accumulate until manually removed. Low operational risk since ~/.claude/orchestration/ is a managed config directory, not a build artifact directory.

2. **Replace --delete with --filter protect rules** — add `--filter='protect *.user.*'` or similar naming-convention guard. Still deletes files that don't match the pattern. Fragile because users must adopt a specific naming convention to be safe. Doesn't satisfy the acceptance criterion cleanly.

3. **Replace rsync with explicit per-file cp loop** — enumerate source files and copy each one, no rsync at all. Eliminates deletion risk but also loses rsync's efficient change detection and verbose output showing which files were updated. Over-engineering for the problem at hand.

4. **Keep --delete, enumerate user files with --exclude** — generate an exclusion list of non-source files at runtime by diffing source vs target. Extremely complex; requires maintaining state about what "source-tracked" means. Not viable for a simple sync script.

## 2. Selected Approach

**Option 1: Remove --delete entirely.**

Rationale: Directly satisfies all three acceptance criteria with a one-line change. The principal risk — stale files accumulating after source deletion — is minor and predictable. The inline comment documents the manual removal procedure explicitly, so operators are not left guessing.
