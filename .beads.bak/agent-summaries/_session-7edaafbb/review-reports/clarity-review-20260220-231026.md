## Findings Catalog

### Finding 1: Comment says "map file format" uses NUL-terminated sentinel but format is tab-delimited newline-separated
- **File(s)**: `scripts/fill-review-slots.sh:178`
- **Severity**: P3
- **Category**: clarity
- **Description**: The comment at line 178 states "values separated by a NUL-terminated sentinel" but the actual map file format uses a tab (`\t`) as the delimiter between slot name and tmpfile path, and each entry is separated by a newline — there is no NUL byte anywhere. The awk code at line 219 confirms this: it uses `index(entry, "\t")` to find the field separator. The "NUL-terminated sentinel" phrase is misleading and describes a format that doesn't exist in the code.
- **Suggested fix**: Replace the comment at line 178 with: `# Map file format (one entry per slot):` `#   <slot_name>\t<tmpfile_path>\n`  — removing the incorrect NUL sentinel mention.
- **Cross-reference**: None

### Finding 2: `TIMED_OUT` variable renamed to `REPORTS_FOUND` but semantics inverted — comment at line 556 uses positive logic inconsistently
- **File(s)**: `orchestration/templates/reviews.md:555-576`
- **Severity**: P3
- **Category**: clarity
- **Description**: The renamed variable `REPORTS_FOUND` (initialized to 0, set to 1 on success) is a positive-sense flag ("found"), but the timeout check at line 577 reads `if [ $REPORTS_FOUND -eq 0 ]` which still tests for the absence of reports. This logic is correct but the inline comment `# set to 1 when all expected reports are present` at line 555 documents the inversion. The old `TIMED_OUT` variable (initialized to 1, set to 0 on success) was negative-sense. The new naming is clearer, but there is no clarifying comment at line 577 explaining why we test `eq 0`. A reader unfamiliar with the history might expect to see `if [ $REPORTS_FOUND -ne 1 ]` for consistency with `REPORTS_FOUND` semantics.
- **Suggested fix**: Add a short inline comment: `if [ $REPORTS_FOUND -eq 0 ]; then  # no reports arrived in time`. This is minimal and removes any ambiguity.
- **Cross-reference**: None

### Finding 3: RULES.md Step 3b-i.5 validation block mixes two different quoting styles inconsistently
- **File(s)**: `orchestration/RULES.md:144-163`
- **Severity**: P3
- **Category**: consistency
- **Description**: The validation code block at Step 3b-i.5 uses `[[ ... ]]` bash conditionals for the CHANGED_FILES check (line 153) but uses POSIX `[ ... ]` for the REVIEW_ROUND check (line 144) and the TASK_IDS check (line 159). The CHANGED_FILES check uses bash-specific `${VAR//[[:space:]]/}` parameter expansion which requires bash anyway, making the inconsistent bracket style needlessly confusing. All three checks in the same code block should use the same style.
- **Suggested fix**: Standardize to `[[ ... ]]` throughout the code block, since the block already relies on bash-specific syntax. Change line 144 `if ! echo "${REVIEW_ROUND}" | grep -qE` to use the same approach consistently, or at minimum add a comment noting the deliberate style difference.
- **Cross-reference**: None

### Finding 4: `extract_agent_section` function behavior changed but function name and docstring not updated

- **Category**: clarity

### Finding 7: `parse-progress-log.sh` "UNREACHABLE" comment could cause confusion — the branch IS reachable in theory
- **File(s)**: `scripts/parse-progress-log.sh:203-206`
- **Severity**: P3
- **Category**: clarity
- **Description**: The new comment at lines 203-206 labels the `if [ -z "$RESUME_STEP" ]` branch as "UNREACHABLE" and explains the invariant. The reasoning is sound for normal execution, but the comment says "This branch can never be reached during normal execution" — the qualifier "during normal execution" implies it CAN be reached during abnormal execution. A future maintainer might reasonably wonder: under what abnormal execution could this happen? If it can't happen at all (e.g., the array is compile-time constant and `SESSION_COMPLETE` is hardcoded in it), the comment should be more definitive. If abnormal conditions could bypass it, that warrants mention.
- **Suggested fix**: Strengthen to: `# UNREACHABLE: RESUME_STEP is always set because SESSION_COMPLETE is in the compile-time` `# constant STEP_KEYS array. If SESSION_COMPLETE were already logged, the exit 2 guard above` `# terminates the script before reaching this loop. No execution path can reach this branch.` Or simply: `# Safety net: cannot be reached — STEP_KEYS always includes SESSION_COMPLETE.`
- **Cross-reference**: None

### Finding 10: `SETUP.md` code fence mismatch — nested markdown code block inside code fence using same backtick count
- **File(s)**: `orchestration/SETUP.md:36-56`
- **Severity**: P3
- **Category**: structure
- **Description**: Lines 36-55 contain a code fence (triple backtick) labeled `markdown` that itself contains a sub-section using triple backticks (lines 46-50). Using the same fence delimiter inside a fence causes rendering breakage in most markdown renderers — the inner ``` at line 46 closes the outer fence prematurely. The outer block is declared at line 38 as ` ```markdown ` and the inner block starts at line 46 with ` ``` `. This means the markdown renderer will show lines 46-50 as non-code text, and lines 51-55 as code. This was not changed in this session but it exists in the scoped file.
- **Suggested fix**: Use 4-tilde fences (`~~~~`) or indent-based code blocks for the inner code block, or use 4+ backticks for the outer fence to allow triple-backtick inner blocks.
- **Cross-reference**: None
