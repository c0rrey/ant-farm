### Finding 1: Inconsistent failure label taxonomy between Pantry and Big Head skeleton

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/templates/pantry.md`:33, `/Users/correy/projects/ant-farm/orchestration/templates/pantry.md`:45, `/Users/correy/projects/ant-farm/orchestration/templates/big-head-skeleton.md`:58
- **Severity**: P3
- **Category**: clarity
- **Description**: The Pantry uses two distinct failure labels -- "INFRASTRUCTURE FAILURE" (Condition 1, line 33) and "SUBSTANCE FAILURE" (Conditions 2-3, lines 45/57) -- to classify different root causes. The Big Head skeleton (line 58) only uses "INFRASTRUCTURE FAILURE" for its missing-report failure artifact. This is correct from a semantic perspective (missing files are infrastructure, not substance), but the labels are never formally defined in one place. A reader encountering "INFRASTRUCTURE FAILURE" in Big Head has no reference to understand it is part of a taxonomy also containing "SUBSTANCE FAILURE." The distinction is clear inside pantry.md because both labels appear together, but it fragments across files.
- **Suggested fix**: Add a brief taxonomy note at the top of the term definitions section in one canonical location (e.g., checkpoints.md or a shared reference) defining "INFRASTRUCTURE FAILURE" vs "SUBSTANCE FAILURE" and what each means. Alternatively, add a one-line comment in big-head-skeleton.md referencing the Pantry's taxonomy.
- **Cross-reference**: None

### Finding 2: Misleading variable name MISSING_REPORTS in polling loop

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/templates/reviews.md`:371
- **Severity**: P3
- **Category**: clarity
- **Description**: The bash variable `MISSING_REPORTS` in the polling loop actually captures the output of `ls` commands that list **found** reports, not missing ones. The `if` condition on line 377 checks `wc -l` equals 4, meaning "all 4 files found." The variable name suggests it holds a list of missing reports, but it holds the opposite -- the list of successfully found files. This inverted naming makes the script logic harder to follow on first read.
- **Suggested fix**: Rename the variable to `FOUND_REPORTS` or `REPORT_CHECK` to match its actual semantics. The condition `if [ "$(echo "$FOUND_REPORTS" | wc -l)" -eq 4 ]` reads naturally as "if we found all 4 reports."
- **Cross-reference**: This is in a code block that Edge Cases might also examine for correctness of the `wc -l` logic.

### Finding 3: Nested fenced code block collision in reviews.md Step 0a error template

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/templates/reviews.md`:414-417
- **Severity**: P3
- **Category**: clarity
- **Description**: The error return template (a markdown code block starting around line 390) contains a nested fenced code block with triple backticks on lines 414-416 (```` ``` / Spawn Big Head again... / ``` ````). In standard markdown, the inner triple backticks will prematurely close the outer code block, causing the remaining lines (417-420) to render as regular text rather than as part of the template. This affects readability when the file is rendered in a markdown viewer.
- **Suggested fix**: Use quadruple backticks (``````) for the outer fence, or indent the inner code block by 4+ spaces instead of using backtick fences, so the nesting is unambiguous.
- **Cross-reference**: None

### Finding 4: Data file format spec inconsistency between skeletons and Pantry output

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/templates/dirt-pusher-skeleton.md`:30, `/Users/correy/projects/ant-farm/orchestration/templates/nitpicker-skeleton.md`:19, `/Users/correy/projects/ant-farm/orchestration/templates/pantry.md`:91-120
- **Severity**: P3
- **Category**: clarity
- **Description**: The dirt-pusher-skeleton.md (line 30) says the data file format is "Sections: Context, Scope Boundaries, Focus." The Pantry (lines 91-120) actually writes a task brief with sections titled "## Context", "## Scope Boundaries", "## Focus", and "## Summary Doc Sections (all required)." The skeleton omits "Summary Doc Sections" from its format hint. Similarly, the nitpicker-skeleton.md (line 19) says "Sections: Scope, Files, Focus, Detailed Instructions" but the Pantry (review mode, lines 229-244) writes briefs with sections including commit range, file list, focus areas, report output path, bead-filing prohibition, messaging guidelines, and full report format -- which don't map cleanly to "Scope, Files, Focus, Detailed Instructions."  The format hints in the skeletons are simplified summaries rather than exact matches of what the Pantry produces, which could confuse agents or template maintainers.
- **Suggested fix**: Either update the skeleton format hints to exactly match the Pantry's actual output section names, or add a note like "(approximate -- see Pantry output for full structure)" so readers know the hint is simplified.
- **Cross-reference**: None

### Finding 5: Failure artifact path collision across Pantry conditions

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/templates/pantry.md`:34, `/Users/correy/projects/ant-farm/orchestration/templates/pantry.md`:48, `/Users/correy/projects/ant-farm/orchestration/templates/pantry.md`:62
- **Severity**: P3
- **Category**: clarity
- **Description**: All three failure conditions in the Pantry (Conditions 1, 2, and 3) write their failure artifacts to the same path: `{session-dir}/prompts/task-{TASK_SUFFIX}-FAILED.md`. If the Pantry encounters Condition 1 for a task, writes the failure artifact, then (in a hypothetical re-run or different failure mode) encounters Condition 2 for the same task, the second artifact would overwrite the first. While this is unlikely in practice (the Pantry halts after the first failure for a given task), the shared path means the failure artifact's content is the only way to distinguish which condition triggered -- there is no path-level differentiation.
- **Suggested fix**: This is acceptable as-is since only one condition can fire per task. Optionally, for maximum auditability, the path could include the condition type (e.g., `task-{TASK_SUFFIX}-FAILED-infra.md` vs `task-{TASK_SUFFIX}-FAILED-substance.md`), but this may be over-engineering. No action required; noting for completeness.
- **Cross-reference**: None

### Finding 6: "Halt and report" vs "skip to the next task" inconsistency in Pantry fail-fast

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/templates/pantry.md`:31, `/Users/correy/projects/ant-farm/orchestration/templates/pantry.md`:43, `/Users/correy/projects/ant-farm/orchestration/templates/pantry.md`:55
- **Severity**: P3
- **Category**: clarity
- **Description**: Line 31 introduces the fail-fast block with "Halt and report for any of these conditions." Condition 1 (line 43) says "Do not proceed with task brief composition for this task" (ambiguous about other tasks). Conditions 2 and 3 (lines 55, 66) explicitly say "skip to the next task." The word "Halt" in the introduction suggests stopping all processing, but the actual behavior is per-task skipping with a partial verdict table returned at the end (lines 68-74). The introductory "Halt" is misleading -- it should say something like "Skip and report" to match the actual per-task behavior.
- **Suggested fix**: Change line 31 from "Halt and report for any of these conditions" to "Skip this task and report for any of these conditions" or "Fail this task and skip to the next for any of these conditions."
- **Cross-reference**: None
