## Findings Catalog

### Finding 1: Scout error metadata file format is minimal compared to success format

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/templates/scout.md`:186-196
- **Severity**: P3
- **Category**: clarity
- **Description**: The success metadata format (lines 68-92) includes 8 fields and 4 sections (Affected Files, Root Cause, Expected Behavior, Acceptance Criteria). The error metadata format (lines 192-196) has only 2 fields (Status and Error Details). While this is intentionally minimal, the contrast is stark enough that a cold agent might wonder if the error file needs more structure. The "example error metadata file" label on line 191 is helpful but the format block lacks the `# Task: {full-task-id}` header's explanation.
- **Suggested fix**: Add a brief comment above the error format: "Error metadata files are intentionally minimal -- the Pantry only needs to detect the error status and skip the task."

### Finding 2: Pantry fail-fast check wording could be clearer about what "does not exist" means

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/templates/pantry.md`:27-31
- **Severity**: P3
- **Category**: clarity
- **Description**: Line 27 says "If the file is missing, does not exist, or contains `**Status**: error`". "Missing" and "does not exist" are synonymous here -- the disjunction reads as three distinct conditions when there are really two (file absent OR file contains error status). This could briefly confuse a cold agent parsing the conditions.
- **Suggested fix**: Simplify to "If the file does not exist or contains `**Status**: error`:"

### Finding 3: Scout Step 2 mode names inconsistent between input and instructions

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/templates/scout.md`:11, 24-30
- **Severity**: P3
- **Category**: clarity
- **Description**: Line 11 declares `**Mode**: {MODE}` as the input, but the mode descriptions on lines 24-30 use backtick-formatted names (`ready`, `epic <epic-id>`, `tasks <id1>, <id2>, ...`, `filter <description>`). The `ready` mode is new (added in commit 55c8401, noted by DMVDC as scope creep outside nr2). The format `{MODE}` is a Tier 1 placeholder but the mode values include embedded arguments (e.g., `epic <epic-id>` is a mode plus a parameter). How the Queen splits `{MODE}` into mode name and argument is not defined.
- **Suggested fix**: Either document the parsing convention (e.g., "the first word is the mode, remaining words are the argument") or split into `{MODE}` and `{MODE_ARG}` placeholders.

### Finding 5: Scout Step 2.5 "Skip files without valid frontmatter" lacks error handling guidance

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/templates/scout.md`:43-44
- **Severity**: P3
- **Category**: clarity
- **Description**: Lines 43-44 say to "read the YAML frontmatter" and "skip files without valid frontmatter." This is clear about what to do, but it does not say whether to log a warning or silently skip. Given that the Error Handling section (lines 184-204) was specifically added in task nr2 to prevent silent dropping, silently skipping agent files with bad frontmatter seems inconsistent with the template's own error handling philosophy.
- **Suggested fix**: Add "Log a warning in the briefing's Errors section for agent files with missing or invalid frontmatter."
- **Cross-reference**: Potential edge-case concern -- an agent file with corrupt frontmatter would be silently excluded from agent recommendations.
