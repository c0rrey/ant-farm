### Finding 6: checkpoints.md verdict vocabulary split between WARN and PARTIAL is inconsistent and operationally dangerous

- **File(s)**: orchestration/templates/checkpoints.md:50, 63-69, 369, 428-430, 549
- **Severity**: P2
- **Category**: clarity
- **Description**: Two distinct problems exist (confirmed by correctness-reviewer investigation):

  **Problem A** (line 50): The preamble states "All checkpoints use three verdict states: PASS, WARN, FAIL" — but DMVDC and CCB use PARTIAL (not WARN) as the repair-and-resubmit state. PARTIAL is correctly defined in "Details by Checkpoint" (lines 84-92), so a complete reader sees the right behavior, but the preamble claim is factually wrong and contradicts the details subsection.

  **Problem B** (lines 68-69): The Checkpoint-Specific Thresholds table says "WARN allows resubmission; FAIL escalates" for DMVDC rows. WARN is not a valid DMVDC verdict. If Pest Control reads the table and outputs "WARN" for a partial DMVDC failure, the Queen's "On PARTIAL or FAIL" handler (lines 443, 567) will not match the string "WARN" — the gap would be silently treated as PASS and the agent would never be resumed.

- **Suggested fix**: Two-line edit: (1) Change "All checkpoints use three verdict states: PASS, WARN, FAIL" to clarify the CCO/WWD vs DMVDC/CCB vocabulary split (e.g., "CCO and WWD use PASS/WARN/FAIL; DMVDC and CCB use PASS/PARTIAL/FAIL"). (2) Replace "WARN allows resubmission" with "PARTIAL allows resubmission" in the DMVDC table rows (lines 68-69).
- **Cross-reference**: Confirmed and supplemented by correctness-reviewer (Finding 5, Group B in correctness report at `.beads/agent-summaries/_session-50c2c6/review-reports/correctness-review-20260219-120000.md`).

### Finding 12: queen-state.md uses `<angle-bracket>` placeholders inconsistently — some have descriptive labels, some are bare

- **File(s)**: orchestration/templates/queen-state.md:1-46 (all lines)
- **Severity**: P3
- **Category**: clarity
- **Description**: The template uses two placeholder styles: `<timestamp>`, `<session-id>`, `<path>`, etc. (descriptive) and bare values like `pending/completed/failed` (options separated by slash). The options style (`pending/completed/failed`) appears in multiple table cells and represents valid values, while `<N>`, `<M>`, `<list>` are fill-in placeholders. This mixing means a reader cannot immediately distinguish "choose one of these values" from "fill in a value here." For example, line 9 has both `pending/completed/failed` (options) and `{path}` (fill-in) in the same row.
- **Suggested fix**: Standardize: use `<angle-bracket>` for fill-in values (e.g., `<N tasks>`) and use a consistent indicator for option lists, such as `[pending|completed|failed]` with brackets and pipe separators, matching common template conventions.
- **Cross-reference**: None

- **Category**: clarity
- **Description**: The round 1 and round 2+ team setup prompts are enclosed in `~~~markdown` fenced blocks (using tildes rather than backticks). This is intentional to avoid conflicting with any inner backtick fences, which is a known Markdown trick. However, the rest of the file uses standard triple-backtick fences.
