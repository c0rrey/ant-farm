## Findings Catalog

### Finding 1: PII scrub reverted by subsequent commit

- **File(s)**: `.beads/issues.jsonl` (all 100 entries)
- **Severity**: P2
- **Category**: clarity (documentation/data hygiene)
- **Description**: Task i0c (commit 047f1be) scrubbed `correycc@gmail.com` from all 100 owner fields. However, commit f455361 ("chore: sync beads JSONL after closing session tasks") re-introduced the email addresses into every record. The current HEAD still contains the PII that i0c was supposed to remove. The README "Forking this repo" section tells adopters to run `bd init` to get a clean slate, but the committed JSONL remains polluted with a personal email address. This is both a clarity issue (the repo claims PII was scrubbed but it was not) and a correctness issue (the acceptance criteria for i0c are no longer met).
- **Suggested fix**: Re-run the PII scrub on issues.jsonl and ensure any future `bd sync` operations preserve the scrub.
- **Cross-reference**: This is also a correctness finding -- messaging correctness reviewer.

### Finding 2: SETUP.md test checklist contradicts Information Diet

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/SETUP.md`:69-78
- **Severity**: P3
- **Category**: clarity
- **Description**: The "Step 3: Test it" section instructs the user to verify that "Claude" runs `bd show`, but the RULES.md Information Diet prohibits the Queen from running bd commands directly. The Scout performs these operations. A new user following this test checklist would expect to see the Queen running `bd show`, which contradicts the documented architecture. (Note: existing issue ant-farm-873 covers the correctness aspect; this finding covers the clarity confusion for new users.)
- **Suggested fix**: Rewrite the test checklist to say "Verify the Scout runs `bd show`" or "Verify task metadata appears in the session directory."

### Finding 3: SETUP.md references hardcoded path to hs_website project

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/SETUP.md`:57, 114
- **Severity**: P3
- **Category**: clarity
- **Description**: Lines 57 and 114 reference `~/projects/hs_website/SESSION_PLAN_TEMPLATE.md` as a source for copying the session template. This is a project-specific path that will not exist for any adopter who is not the original developer. A fork adopter following this guide will encounter a missing file with no explanation.
- **Suggested fix**: Either include a SESSION_PLAN_TEMPLATE.md in the ant-farm repo itself, or replace the path with a generic instruction like "Copy from a reference project or create from the template below."

### Finding 4: README "Forking this repo" hook install step is incomplete

- **File(s)**: `/Users/correy/projects/ant-farm/README.md`:294-297
- **Severity**: P3
- **Category**: clarity
- **Description**: Step 4 of the "Steps for new adopters" section says to install git hooks but only provides a comment (`# See orchestration/SETUP.md for full hook installation instructions`) instead of the actual command. Steps 2 and 3 provide inline commands, making step 4 inconsistent and forcing the reader to jump to another document for a single command.
- **Suggested fix**: Replace the comment with the actual command: `./scripts/install-hooks.sh`

### Finding 5: SETUP.md nested code fences break markdown rendering

- **File(s)**: `/Users/correy/projects/ant-farm/orchestration/SETUP.md`:34-51
- **Severity**: P3
- **Category**: clarity
- **Description**: The "Recipe Card" and "Quick Setup" sections embed code blocks (triple backtick fences) inside outer code blocks. In standard markdown renderers, the inner fences will close the outer block prematurely, producing mangled output. The "Add this section" block starting at line 34 contains nested ` ```markdown ` and ` ``` ` fences that will not render correctly.
- **Suggested fix**: Use a different fence style for the outer block (e.g., `~~~` or indent-based blocks), or escape the inner fences.

### Finding 6: install-hooks.sh backup does not use unique filenames

- **File(s)**: `/Users/correy/projects/ant-farm/scripts/install-hooks.sh`:25-27
- **Severity**: P3
- **Category**: clarity
- **Description**: When backing up an existing pre-push hook, the script uses a fixed name `$HOOK_TARGET.bak`. If the installer is run multiple times, each run overwrites the previous backup. Compare with sync-to-claude.sh which uses a timestamped backup name (`CLAUDE.md.bak.$(date +%Y%m%dT%H%M%S)`). The inconsistency is confusing and the fixed name means only the most recent backup survives.
- **Suggested fix**: Use a timestamped backup name consistent with sync-to-claude.sh: `BACKUP="$HOOK_TARGET.bak.$(date +%Y%m%dT%H%M%S)"`

### Finding 7: sync-to-claude.sh missing shebang explanation or inline comment for rsync --delete

- **File(s)**: `/Users/correy/projects/ant-farm/scripts/sync-to-claude.sh`:23
- **Severity**: P3
- **Category**: clarity
- **Description**: The `rsync -av --delete` command will remove files from `~/.claude/orchestration/` that no longer exist in the source. While the inline comment says "--delete removes files from target that no longer exist in source," the implications are not fully explained. A user who has placed custom files in `~/.claude/orchestration/` (outside this repo) will lose them silently on next push. The backup mechanism only covers CLAUDE.md, not the orchestration directory.
- **Suggested fix**: Add a brief comment noting that `--delete` means custom files in the target will be removed, or add a warning to SETUP.md about this behavior.
