## Findings Catalog

### Finding 1: Inconsistent team name — "nitpickers" vs "nitpicker-team"
- **File(s)**: `orchestration/templates/reviews.md:982`
- **Severity**: P2
- **Category**: clarity
- **Description**: The new Fix Workflow text added in this diff uses `team_name: "nitpickers"` (L982 of the current file), while `orchestration/RULES.md` (L319, L339, L343, etc.) consistently uses `team_name: "nitpicker-team"`. A reader implementing the fix workflow from `reviews.md` would use the wrong team name and the Task call would fail silently or target a non-existent team. This is a naming inconsistency introduced in the same change set.
- **Suggested fix**: Change `team_name: "nitpickers"` at `reviews.md:982` to `team_name: "nitpicker-team"` to match the authoritative name used throughout RULES.md.
- **Cross-reference**: This also qualifies as a Drift issue (stale cross-file reference). Deferring report to drift-reviewer for authoritative filing — see Cross-Review Messages.

### Finding 2: "Fix handoff" label on a round-transition message misleads about timing
- **File(s)**: `orchestration/templates/big-head-skeleton.md:187`
- **Severity**: P3
- **Category**: clarity
- **Description**: Step 12's opening phrase is "Fix handoff: <N> root causes filed." — but this message is sent to the Queen, not to fix agents, and it fires before any fixes happen. The word "handoff" implies transferring work to fixers, but the message is actually a summary of what was filed and initiates the triage phase. The "Fix handoff" label appears at the top of the `SendMessage` format block, which a reader might interpret as a message type/subject line.
- **Suggested fix**: Rename the message header to "Review complete: <N> root causes filed." or "Bead filing complete: <N> root causes filed." to make the timing and direction clear.

### Finding 3: "Team roster progression" bullet for "After fix wave" is dense and hard to parse
- **File(s)**: `orchestration/RULES.md:168`
- **Severity**: P3
- **Category**: clarity
- **Description**: The second bullet in the new Team roster progression block reads: `+ N fix DPs + fix-pc-wwd + fix-pc-dmvdc (names: fix-dp-1..N, fix-pc-wwd, fix-pc-dmvdc; round suffixes for round 2+: fix-dp-r2-1, fix-pc-wwd-r2, fix-pc-dmvdc-r2)`. This packs names, counts, and round-suffix rules into a single parenthetical. A developer scanning this block to understand team membership would need to read it twice to extract structure.
- **Suggested fix**: Split the parenthetical into two sub-bullets: one listing the names added per wave, and a second noting the round-suffix convention. Alternatively, point to the Fix Team Member Naming table in reviews.md rather than duplicating the information here.

### Finding 4: Step 3b-v dummy reviewer prompt hard-codes `<N>` placeholder instead of actual round
- **File(s)**: `orchestration/RULES.md:257`
- **Severity**: P3
- **Category**: clarity
- **Description**: The tmux window name `DUMMY_WINDOW="dummy-reviewer-round-<N>"` uses an angle-bracket placeholder `<N>` in the variable assignment, but the Notes section says "Replace `<N>` in `DUMMY_WINDOW` with the actual round number." This instruction relies on a developer remembering to substitute the placeholder before running the shell block. The surrounding code uses `${REVIEW_ROUND}` as a shell variable — using it here too would make the code self-contained and eliminate the manual substitution requirement.
- **Suggested fix**: Change the literal `<N>` to `${REVIEW_ROUND}` in the DUMMY_WINDOW assignment to match the variable-substitution style used in the rest of the block. Remove the corresponding Note if the note becomes redundant.

### Finding 5: "Information Diet" section header contradicts its own opening sentence
- **File(s)**: `orchestration/RULES.md:494`
- **Severity**: P3
- **Category**: clarity
- **Description**: The section header is "Information Diet (The Queen's Window)" and the opening sentence says "The Queen's read permissions are defined explicitly and authoritatively in the 'Queen Read Permissions' section above. **Authoritative source: Queen Read Permissions above.**" The next sentence calls this section a "Quick reference." The self-description as both a section header in its own right and a "Quick reference" pointing elsewhere creates ambiguity about whether a reader should rely on this section or scroll up. The section exists to help mid-document orientation but its authority is ambiguous.
- **Suggested fix**: Change the section header from "Information Diet (The Queen's Window)" to "Information Diet — Quick Reference (non-authoritative)" and remove the bold "Authoritative source:" label, since the header already signals the section's role.

### Finding 6: reviews.md "Round Transition via SendMessage" references a stale path `{session-dir}/briefs/review-brief-<timestamp>.md`
- **File(s)**: `orchestration/templates/reviews.md:1091`
- **Severity**: P3
- **Category**: clarity
- **Description**: Step 1 in the Round Transition section instructs the Queen to include `Brief path: {session-dir}/briefs/review-brief-<timestamp>.md` in the SendMessage to the Correctness reviewer. The briefs directory is not created at Step 0 (RULES.md L572 creates `task-metadata`, `previews`, `prompts`, `pc`, `summaries` — not `briefs`). This path appears to be a copy from an earlier workflow design and no longer matches the session directory layout. A developer following this instruction would construct a path that does not exist.
- **Suggested fix**: Determine whether this field is needed and, if so, replace the path with the correct artifact location (e.g., `{session-dir}/prompts/review-correctness.md` from Step 3b-ii). If the field is unused, remove it from the SendMessage template.
- **Cross-reference**: This overlaps with Drift (stale path reference). Filing here as Clarity because the immediate problem is that the instruction is confusing to follow; also messaging drift-reviewer.

### Finding 7: checkpoints.md CCB section — Check 3b numbering creates a non-standard gap
- **File(s)**: `orchestration/templates/checkpoints.md:568`
- **Severity**: P3
- **Category**: clarity
- **Description**: CCB checks are numbered 0, 1, 2, 3, 3b, 4, 5, 6, 7. The "3b" label is unusual — it breaks the reader's expectation of sequential integers. The "b" suffix suggests it was inserted after-the-fact between Check 3 and Check 4. While the naming is explained by insertion order, a new reader would be uncertain whether "3b" is a sub-check of 3 or a peer check at the same level.
- **Suggested fix**: Renumber the checks sequentially (0 through 8, making the current Check 3b the new Check 4 and shifting all following checks up by 1). Update all references to check numbers in the same file. If the Verdict Thresholds Summary table references "All 8 checks" it would need updating to "All 9 checks" — verify.
