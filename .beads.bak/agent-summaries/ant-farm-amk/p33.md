## Approaches Considered

### Approach A: Strict Uppercase Convention (Queen-Substitution Model)
**Strategy**: All placeholders meant for Queen substitution use `{UPPERCASE}`. Shell variables use `${UPPERCASE}`. Lowercase `{lowercase}` reserved for output examples only (not substitution).

**Pros**:
- Clear visual distinction between Queen-substituted and example content
- Aligns with existing orchestration template usage
- Easy grep pattern: `\{[A-Z_]+\}` finds all real placeholders
- Shell variables clearly distinct with `${}` syntax

**Cons**:
- Some files already use lowercase in agent-facing context (e.g., scout.md `{session-dir}`)
- Would require refactoring existing files
- Doesn't capture semantic difference between "runtime-derived" and "output example"

**Risks**: Breaking changes to existing templates; migration overhead

---

### Approach B: Flexible Casing with Explicit Term Definitions
**Strategy**: Allow both uppercase and lowercase. Every file using placeholders must define a term block explicitly stating whether each placeholder is "Queen-substituted", "agent-derived", or "output example".

**Pros**:
- Accommodates existing inconsistencies without refactoring
- Self-documenting (each file defines its placeholders)
- More forgiving of accidental casing

**Cons**:
- Requires term definition blocks in all files (verbose)
- Harder to grep — no single pattern captures all real placeholders
- Multiple sources of truth (term definitions vs actual usage)
- Does not solve semantic ambiguity about intent

**Risks**: Easy to forget to update term definition when adding new placeholder

---

### Approach C: Tiered Placeholder System (SELECTED)
**Strategy**: Three distinct syntaxes map to three workflow phases:
- `{UPPERCASE}` = Queen-substituted before spawn
- `{lowercase-kebab}` = agent-derived at runtime or output format example
- `${SHELL_VAR}` = bash/shell variables in code blocks only

**Pros**:
- Semantically rich — each syntax conveys intent AND timing
- Enables grep patterns per tier: `\{[A-Z]+\}`, `\{[a-z-]+\}`, `\$\{[A-Z_]+\}`
- Clear mental model: "uppercase = pre-spawn", "lowercase = post-spawn or example", "shell = bash variables"
- Matches existing best practices already in use across templates
- Requires zero refactoring (current usage is already compliant!)

**Cons**:
- Three patterns instead of one to remember
- Requires documentation (hence this convention document)

**Risks**: Low — convention documents existing correct practice rather than imposing change

---

### Approach D: Minimal Canonical Reference + Audit
**Strategy**: Create reference document without enforcing casing. Document current mixed usage pattern and justify each inconsistency.

**Pros**:
- Minimal effort
- Acknowledges reality of existing usage

**Cons**:
- Does not fix inconsistencies
- Grep audit still shows "violations" even if justified
- Future authors still have ambiguity
- Codifies technical debt

**Risks**: Perpetuates confusion; future adoption remains unclear

---

## Selected Approach: Approach C - Tiered Placeholder System

**Rationale**:

1. **Semantic Clarity**: Three distinct placeholders map naturally to three workflow phases:
   - Pre-spawn (Queen fills in): `{UPPERCASE}`
   - Runtime (agent computes): `{lowercase}`
   - Shell execution: `${SHELL_VAR}`

2. **Matches Existing Best Practices**: Audit found that the development team **already uses this convention correctly** throughout all templates. The convention documents and formalizes what was unconsciously implemented:
   - Queen provides `{SESSION_DIR}` before spawn
   - Agent uses it to construct `{session-dir}` references in output
   - RULES.md correctly uses `${SESSION_DIR}` in bash code block

3. **Zero Refactoring Required**: All 14 files in orchestration/ already comply. No changes needed. This dramatically lowers risk and implementation effort.

4. **Enables Mechanical Validation**: Clear grep patterns per tier make automated auditing possible. Acceptance criteria #3 (grep validation) is now testable.

5. **Future-Proofs System**: New template authors can see intent by casing alone, reducing confusion and increasing adoption.
