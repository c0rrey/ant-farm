## 1. Approaches Considered

### Approach A: Sequential single-pass edit, bottom-to-top
Make all edits in reverse line-number order so each edit does not disturb the line numbers of subsequent edits. Requires careful tracking of insertion offsets across 13 modification points.
**Tradeoff**: Bottom-to-top ordering is unnatural to reason about; error-prone when insertion sizes vary; hard to verify the final state without re-reading the entire file after each change.

### Approach B: Full-file rewrite
Read the entire RULES.md, compose the updated version in full, and overwrite the file in a single Write call. Guarantees no leftover content from the old version and produces a single atomic change.
**Tradeoff**: The entire 500-line file must be reproduced verbatim. Any transcription error in unchanged sections introduces unintended modifications. The diff becomes large and harder to review for adjacent regressions.

### Approach C: Targeted Edit calls, top to bottom (SELECTED)
Use the Edit tool's `old_string`/`new_string` mechanism targeting unique surrounding context. Work top to bottom since the Edit tool matches content rather than line numbers. Each change is isolated and independently verifiable.
**Tradeoff**: Most surgical approach — each of the 11 Edit calls touches only the intended text. Requires unique context strings for each replacement; requires re-reading the file for verification after all edits complete.

### Approach D: Sectional block replacement
Group logically adjacent changes into 5-6 larger Edit blocks (e.g., Steps 4+5+5b+5c+6 as one block; tables as separate blocks). Reduces the number of Edit tool calls.
**Tradeoff**: Larger replacement blocks are harder to diff mentally between the old and new content. Risk of inadvertently altering content between the targeted points within each section. Does not improve accuracy relative to Approach C.

## 2. Selected Approach with Rationale

**Approach C — targeted Edit calls, top to bottom.**

The 13 modification points touch 7 distinct sections across ~450 lines. Approach C isolates each change to its minimum necessary context, making each edit verifiable on its own merits. The Edit tool's content-matching (not line-number-matching) makes top-to-bottom ordering safe: prior insertions shift line numbers but do not break subsequent content-based matches. The result is a precise, auditable set of changes with no risk of transcription errors in unchanged sections.
