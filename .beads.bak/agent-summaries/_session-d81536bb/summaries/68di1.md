## 1. Approaches Considered

**Approach A — Pure prose instructions**
Write all Scribe instructions as flowing paragraphs with no tables or structured lists. Maximizes LLM reading comprehension for plain-text models. Tradeoff: poor scannability for the Queen filling placeholders; hard to verify completeness of 7 input sources at a glance.

**Approach B — Minimal skeleton mirroring dirt-pusher exactly**
Copy dirt-pusher-skeleton.md's exact structure and length, keeping agent-facing text as brief as possible. Maximizes structural consistency with existing templates. Tradeoff: insufficient guidance for a complex multi-source, multi-output task — the Scribe must coordinate 7 reads and 2 distinct write operations with derivation rules. A minimal prompt increases hallucination risk and inconsistency across sessions.

**Approach C — Rich structured template with explicit sections per step (SELECTED)**
Use the dirt-pusher's Queen-instructions header and Step 0 pattern verbatim, but expand the agent-facing body into clearly headed steps with embedded format examples. Include inline count-derivation rules and edge-case handling (empty open-bead list, no review rounds). Tradeoff: slightly longer token count when sent to the Scribe, but dramatically reduces ambiguity. Given the Scribe runs once per session and correctness is verified by ESV, optimizing for reliability over token economy is the right call.

**Approach D — Data-driven table-first design**
Open the agent-facing section with a sources table and an outputs table, then list derivation rules as numbered items at the bottom. Token-efficient and clean. Tradeoff: tables feel like reference material rather than instructions to execute in order; risks the agent skimming the tables and missing procedural nuances like the CHANGELOG prepend protocol or the empty-bead-ID guard.

---

## 2. Selected Approach with Rationale

Approach C was selected because the Scribe task has meaningfully higher complexity than a dirt-pusher: 7 input sources, 2 outputs with distinct content rules, a duration math instruction, and a CHANGELOG format that must precisely match an existing convention. Rich structured guidance with embedded format examples minimizes variance across sessions. The dirt-pusher pattern is preserved for the Queen-facing header block and Step 0 (lines 28-31 of dirt-pusher-skeleton.md), ensuring structural consistency across all skeleton templates.
