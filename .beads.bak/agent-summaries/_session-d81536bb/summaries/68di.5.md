## 1. Approaches Considered

### Approach A — Surgical line-level edits (Selected)
Edit only the exact lines that reference "Step 4 CHANGELOG" across each file. Each change is a narrow string replacement targeting specific lines identified by the task brief. Minimal diff, easy peer review, zero risk of disrupting surrounding content or formatting. No prose surrounding the changed lines is altered.

**Tradeoffs**: Produces the smallest possible diff. Does not improve surrounding prose that might also be stale. Risk of leaving the pattern `Step 4.*CHANGELOG` on the same line if the note about Scribe is appended inline rather than on a separate line (discovered during implementation and corrected).

### Approach B — Section-level rewrite
Rewrite each entire containing section to integrate the new Scribe/ESV description more naturally. For example, the "Termination Rule" block in reviews.md would be rewritten end-to-end with improved flow, rather than just patching line 194.

**Tradeoffs**: Produces more readable prose and integrates the new workflow more organically. However, the diff is significantly larger, increasing review burden and the risk of introducing unintended changes. Violates the task's "only update cross-references" scope constraint.

### Approach C — Regex search-and-replace across all files
Write a sed/awk substitution to globally replace "Step 4" + CHANGELOG patterns in all template files. Fast for bulk changes.

**Tradeoffs**: Risks touching files outside scope (e.g., _archive/ files, RULES.md which is explicitly off-limits). Regex patterns for this context-sensitive replacement are fragile — "Step 4" appears many times legitimately in contexts unrelated to CHANGELOG. High false-positive risk. Requires careful exclusion lists. Not appropriate for the small number of targeted changes needed.

### Approach D — Full-file read-modify-write
Read each entire file, make changes, and write the complete file back using the Write tool. Avoids the need for precise string matching in the Edit tool.

**Tradeoffs**: Unnecessary for targeted edits and risks accidentally dropping content at file boundaries. The Edit tool is precise and appropriate for these changes. Full-file replacement also makes the diff harder to read in git history.
