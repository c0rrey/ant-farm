## 1. Approaches Considered

**Approach A — Sequential bead-by-bead, single-file focus**
Read RULES.md once, then evaluate each bead against that snapshot in order. Simple and traceable but does not leverage git history to detect recent fixes, risking false STILL_VALID calls on already-fixed items.

**Approach B — Git-log-first triage**
Search git log for commits referencing each bead ID. Any bead with a named fix commit gets tentatively ALREADY_FIXED, then verify current file state. Fast for known fixes but misses cases where a fix was committed under a different bead ID (collateral fixes).

**Approach C — Current-state grep pass**
For each bead, grep the current file for the specific text the bead says is wrong. If the text is absent, tentatively ALREADY_FIXED; if present, STILL_VALID. Efficient for textual issues but unreliable for structural issues (e.g., a missing table row cannot be detected by grep alone).

**Approach D — Hybrid: git log + current-state verification per bead**
Scan git log for relevant commits (collateral or direct), then confirm current file state for each bead. For beads about files other than RULES.md, use targeted grep to verify the primary symptom. Used as the triage accelerator for ALREADY_FIXED candidates; full evidence check for all verdicts regardless.

## 2. Selected Approach

Rationale: The git log contains named commits (e.g., 5dba086 for ant-farm-ovw8, d4aa294 for ant-farm-88zh) that directly fixed issues adjacent to beads in this batch.
