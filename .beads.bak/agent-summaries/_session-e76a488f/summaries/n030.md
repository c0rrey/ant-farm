## 1. Approaches Considered

**Approach A: Sequential file-by-file scan**
Read each referenced file, then iterate all 15 beads in input order, checking each against whichever file it references. Simple but requires holding all file content in context simultaneously.
- Pro: Predictable order, no grouping overhead.
- Con: Redundant re-reads if multiple beads reference the same file passage.

**Approach B: Batch by referenced file**
Group the 15 beads by the file they reference (README.md group, SETUP.md group, MEMORY.md group, SESSION_PLAN_TEMPLATE group), verify each group together after reading that file once.
- Pro: Efficient — one read per file, all related beads checked together.
- Con: Requires upfront grouping; output must still be in bead-ID order.

**Approach C: Duplicates-first heuristic**
Handle the 4 known duplicate pairs first (8 beads auto-resolved), then verify the remaining 7 against source files.
- Pro: Reduces the investigation surface quickly.
- Con: Risk of rubber-stamping duplicates without verifying the underlying issue is genuinely present; less rigorous.

**Approach D: Evidence-driven per-bead analysis** (SELECTED)
For each bead, locate the exact passage in the source file that the bead claims is wrong, record the line number and text, then assign a verdict based on whether the issue is present, absent (fixed), or duplicated.
- Pro: Produces the highest-quality evidence strings; satisfies the acceptance criterion that every ALREADY_FIXED verdict must cite specific evidence. Naturally handles duplicates because duplicate pairs produce identical evidence passages.
- Con: Slightly more careful work than batch-grouping, but produces more reliable output.

## 2. Selected Approach with Rationale

Selected **Approach D (Evidence-driven per-bead analysis)** because:
- Acceptance criterion 4 requires specific evidence for every ALREADY_FIXED verdict.
- Acceptance criterion 3 requires all 4 known duplicate pairs to be marked DUPLICATE_SUSPECT — verifying the underlying issue for each pair (rather than just matching titles) provides higher confidence.
- The MEMORY.md constraint (criterion 5) requires case-by-case judgment about what "still valid" means for a user-maintained file, which is best handled bead-by-bead rather than as a batch.
