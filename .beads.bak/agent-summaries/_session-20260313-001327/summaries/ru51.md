## Files Changed

- `crumb.py` — fixed dual-lookup for `parent` and `discovered_from` in `cmd_list` filters and `_auto_close_trail_if_complete`

## Implementation

Three code paths were updated to check both top-level fields and the `links` sub-dict, following the pattern already established in `_get_trail_children` (L411-412), `cmd_doctor` (L1759-1762), and `_get_blocked_by` (L861):

1. **`cmd_list --parent` filter (L530-535)**: Changed from `t.get("parent") == args.parent` to an `or` condition that also checks `(t.get("links") or {}).get("parent") == args.parent`. The `or {}` guard handles the case where `links` is absent or `None`.

2. **`cmd_list --discovered` filter (L537-541)**: Changed from `t.get("discovered_from")` (truthy check) to also check `(t.get("links") or {}).get("discovered_from")`. The filter checks for presence of any value, not equality, so both branches use a truthy test.

3. **`_auto_close_trail_if_complete` parent lookup (L436-439)**: Changed `parent_id = links.get("parent")` to `parent_id = crumb.get("parent") or links.get("parent")`. Also tightened the `isinstance` guard — previously the function returned early if `links` wasn't a dict, which would have silently skipped auto-close for crumbs with a top-level `parent` but malformed `links`. Now it reassigns `links = {}` and falls through to the top-level check.

## Approaches Considered

1. **Extract a `get_linked_field(record, field)` helper**: The bug description explicitly suggests this. It would DRY up the pattern across all six call sites. Rejected for this fix because the three existing correct sites (`_get_trail_children`, `cmd_doctor`, `_get_blocked_by`) use slightly different lookup patterns and refactoring all six would expand scope significantly. The inline `or`-expression is consistent with the existing pattern at L411-412.

2. **Normalize fields at load time in `read_tasks()`**: Canonicalize all records to always have top-level fields by copying from `links` on read. This would eliminate the dual-lookup problem entirely. Rejected because it mutates records in place at load time, risks losing round-trip fidelity (a record written back would gain fields it didn't originally have), and represents a larger data-model change outside the scope of this bug.

3. **Use a single canonical location (`links` always wins)**: Deprecate top-level fields entirely and always read from `links`. Rejected because `cmd_doctor` and other existing code already checks both, and existing task files have records with top-level fields that would silently lose their parent linkage.

4. **Change `_auto_close_trail_if_complete` to call `_get_trail_children` from the trail's perspective instead of reading the child's parent**: `_get_trail_children` already does dual-lookup (it checks `links.parent`). If we identified the trail first then asked for its children, we'd avoid the parent-lookup problem entirely. Rejected because the function receives a `closed_crumb_id`, not a trail ID — we'd still need to find the parent first to know which trail to query, and we'd be doing more work than the simple dual-field read.

5. **Add a separate `parent` key normalization step in `crumb link`**: Ensure `crumb link --parent` always writes to both locations. Rejected because this fixes the write path, not the read path, and records already in the wild with only a top-level `parent` field would remain broken.

## Per-File Correctness Notes

### crumb.py

- **`cmd_list --parent` (L530-535)**: The `(t.get("links") or {})` guard correctly handles `None`, missing key, and empty dict — all evaluate to `{}` so `.get("parent")` returns `None` safely. No false positives: `None == args.parent` is always false when `args.parent` is a non-empty string.
- **`cmd_list --discovered` (L537-541)**: Truthy check is correct here — the feature is "was this crumb discovered from anywhere", not equality with a specific ID. Both the top-level and `links` branches use the same truthy semantics.
- **`_auto_close_trail_if_complete` (L436-439)**: `crumb.get("parent") or links.get("parent")` uses short-circuit evaluation — if top-level `parent` is a non-empty string, `links.get("parent")` is never evaluated. If top-level `parent` is absent/`None`/empty string, falls through to `links`. This mirrors the semantics of `_get_trail_children` which checks `links.parent` exclusively — there is no case where both locations hold conflicting parent IDs for the same record, so `or` is safe.
- The `isinstance` guard change (from `return` to `links = {}`): previously a crumb with a non-dict `links` value and a top-level `parent` field would cause silent no-op on auto-close. Now it continues to the `parent_id` assignment and auto-close proceeds correctly.
