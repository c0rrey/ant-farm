## Files Changed

- `crumb.py` — wrapped unguarded `open()` and `path.touch()` calls in `try/except OSError` blocks

## Implementation

Three locations were fixed following the pattern established in `read_config()` at L139:

1. **`read_tasks()` (L181)**: Replaced `with open(path, "r") as fh:` with a split open-then-context-manager pattern: `open()` is called first inside a `try/except OSError`, storing the file handle in `fh_ctx`, which is then used in the `with` statement. This separates the OS-level open from the iteration logic cleanly.

2. **`iter_jsonl()` (L222)**: Same split pattern applied — `open()` in `try/except OSError as exc: die(...)`, then `with fh_ctx as fh:` for iteration. This function is currently unused but the fix eliminates latent risk.

3. **`FileLock.__enter__()` (L258-260)**: Wrapped both `path.touch()` and `open(path, "w")` inside a single `try/except OSError as exc: die(f"cannot acquire lock: {exc}")` block, since both operations are part of the same lock-acquisition step and either failing means the lock cannot be acquired.

## Approaches Considered

1. **Inline `try/except` wrapping the entire `with` block**: Wrapping `with open(...) as fh: ...` in `try/except OSError` would catch errors from the open itself but also conflate OS errors from inside the loop (e.g., read errors on a network file). The split pattern is more precise — it catches only the open() failure.

2. **Helper function `safe_open()`**: A utility like `def safe_open(path, mode): try: return open(...) except OSError as exc: die(...)` would DRY up the pattern across all three sites. Rejected because it adds indirection and the codebase has only three call sites — premature abstraction. The inline pattern matches `read_config()` which the bug description explicitly cites as the reference.

3. **Re-raise as a custom exception instead of `die()`**: Could raise a custom `CrumbError` so callers can decide how to handle it. Rejected because `die()` is the established convention throughout crumb.py for fatal user-visible errors, and all three sites are indeed fatal — there's no meaningful recovery from being unable to open the tasks file or lock file.

4. **Use `Path.open()` instead of `open()`**: `path.open("r")` is equivalent and slightly more idiomatic with pathlib. Not adopted because the existing codebase mixes both styles and the bug description references `open()` specifically; consistency within the affected functions was more important than cross-file style.

5. **`contextlib.suppress` or two-phase check**: Using `if not path.exists(): die(...)` before opening avoids try/except syntax. Rejected because this is a TOCTOU race — the file can disappear or permissions can change between the check and the open. The bug description explicitly calls this out as one of the failure modes being fixed.

## Per-File Correctness Notes

### crumb.py

- `read_tasks()`: The `fh_ctx` variable is always assigned before the `with` block because `die()` calls `sys.exit()`, so if the `except` branch runs, execution never reaches `with fh_ctx`. No unbound variable risk.
- `iter_jsonl()`: Same guarantee — `die()` terminates before the `with fh_ctx` line. The generator's lazy evaluation does not affect this; the `open()` call is at the top of the generator body and executes on first `next()` call.
- `FileLock.__enter__()`: Both `path.touch()` and `open(path, "w")` share one `try/except` block. If `touch()` succeeds but `open()` raises, the empty lock file is left on disk — this is acceptable because the file being present without a lock holder is harmless; the next caller will succeed normally. `self._lock_file` is not set in the except branch so `__exit__` safely no-ops (checks `if self._lock_file is not None`).
