### F1 — Redundant wrapper function `crumbs_dir` obscures naming
**File**: `/Users/correy/projects/ant-farm/crumb.py:105-107`
**Severity**: P3
**Category**: Naming / Structure

`crumbs_dir()` is a one-line delegate to `find_crumbs_dir()`. The two names suggest different purposes, but both functions do the same thing. A reader encountering `crumbs_dir()` at a call site must trace back to discover it just calls `find_crumbs_dir()`, adding indirection with no payoff.

**Suggested fix**: Inline `crumbs_dir()` call sites to use `find_crumbs_dir()` directly, and remove the wrapper. Or keep only `crumbs_dir` and fold the walk-up logic into it.

### F4 — `--parent` filter in `cmd_list` checks only top-level field; dual-location pattern undocumented
**File**: `/Users/correy/projects/ant-farm/crumb.py:519-520`
**Severity**: P2
**Category**: Documentation / Consistency

The `--parent` filter at line 520 checks `t.get("parent")` (top-level field) but the codebase stores parent linkage in `links.parent` (set by `cmd_link`, read by `_get_trail_children`). The `_get_blocked_by` function at line 861 explicitly documents the dual-location pattern for `blocked_by`, but no equivalent note exists near the `--parent` filter. A developer extending the filter would see what looks like a complete implementation and miss the second storage location.

**Cross-domain note**: The incomplete filter is a potential correctness concern — forwarded to correctness-reviewer separately.

**Suggested fix (clarity)**: Add a comment at line 519 noting that parent may also be stored under `links.parent`, and that both locations should be checked (mirroring the `_get_blocked_by` docstring pattern at line 861).

### F5 — `getattr` with default on always-present argparse fields is misleading
**File**: `/Users/correy/projects/ant-farm/crumb.py:1558`, `1792`
**Severity**: P3
**Category**: Naming / Style

`getattr(args, "from_beads", False)` and `getattr(args, "fix", False)` use a safety default that implies the attribute might be absent. Both fields are unconditionally registered by their respective argparse subparsers with `action="store_true"`, so they are always `False` if not supplied. Direct attribute access (`args.from_beads`, `args.fix`) communicates the correct semantics.

**Suggested fix**: Replace both `getattr(args, ...)` calls with direct attribute access.
