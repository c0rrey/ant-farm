## 1. Approaches Considered

**Approach A — Targeted line-by-line edits (Edit tool per change)**
Apply one Edit call per semantic change (prohibition line, code block, path reference) repeated for each file. Gives a granular audit trail. Risk: easy to miss a token inside a fenced code block; the identical structure of both files makes it straightforward to get confused about which file is in scope for each call.

**Approach B — Full file rewrite (Write tool)**
Read each file, construct the corrected content in full, write it back. Since both files are 81 lines and completely identical, a single mental model of the diff applies to both. No risk of residual tokens surviving inside code fences. Small files make full-content review practical.

**Approach C — Bash regex substitution (sed)**
Use `sed -i` with a pattern list to perform all substitutions in one pass per file. Fast for large files but less controllable — sed word-boundary behaviour on macOS BSD sed differs from GNU sed, and inline-editing of system-level files via shell one-liners is harder to audit.

**Approach D — Edit tool with replace_all flag**
Use the Edit tool's `replace_all: true` parameter to hit every instance of a pattern across a file. Appropriate when a string appears many times and the surrounding context differs. For this task each changed string appears exactly once, making `replace_all` no different from a standard Edit call — and it still requires multiple calls for the three distinct change types.

## 2. Selected Approach

**Approach B (full file rewrite)** was selected.

Rationale: Both files are short (81 lines), identical in content, and require three distinct types of changes (prohibition list, code block removal, path/terminology). Writing the corrected file in full eliminates the risk of a missed token surviving inside a fenced code block — the most likely failure mode with targeted edits. The complete content can be verified in a single Read pass, which satisfies the mandatory review step cleanly.
