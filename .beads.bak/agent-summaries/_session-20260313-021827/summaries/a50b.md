## 1. Approaches Considered

**Approach A — Line-by-line micro-edits**: One Edit call per `bd` command occurrence. Maximally precise and each substitution is independently verifiable, but produces many small diffs that are harder to review as a unit and risk leaving surrounding prose inconsistent.

**Approach B — Wholesale section replacement**: Replace the entire workflow block (L113-190) as a single large old/new string. Guarantees no missed lines but the matching string is fragile — any whitespace deviation causes the edit to fail, and a large replacement is hard to verify for correctness post-edit.

**Approach C — Targeted replacements by logical section (selected)**: Three edits aligned to the three code fences and their surrounding prose: (1) the cross-session dedup bash block + adjacent prose, (2) the P1/P2 bead-filing bash block, (3) the P3 auto-filing bash block plus the `bd epic create` line preceding it. Balances precision with reviewability and matches the natural section boundaries already present in the file.

**Approach D — Pre-process with sed, then review**: Run `sed` substitutions in bash to handle high-frequency patterns (`bd list`, `bd create`) uniformly, then hand-edit structural differences (`bd dep add`, `bd epic create`). Reduces repetition risk but makes the diff non-atomic and harder to trace in version history.

## 2. Selected Approach with Rationale

Approach C was selected. The three code fences in L114-185 are the natural units of change: they are visually and semantically distinct, each maps to exactly one command category (dedup check, P1/P2 filing, P3 filing). Replacing them individually, together with the immediately adjacent prose that references the old command names, keeps each edit small enough to verify in one reading and large enough to capture all related terminology changes (e.g., "bead" to "crumb" in error messages and log strings).
