### Finding C-04
- **File:line**: `orchestration/RULES.md:383`
- **Severity**: P3
- **Category**: Magic value — unexplained literal
- **Description**: The session ID generation command `SESSION_ID=$(echo "$$-$(date +%s)-$RANDOM" | shasum | head -c 8)` uses `head -c 8` to truncate the SHA hash. The number `8` is a magic value with no comment explaining why 8 hex chars (32 bits) is sufficient uniqueness for session IDs in this context. A reader maintaining this line cannot evaluate whether changing it to 6 or 12 would be safe.
- **Suggested fix**: Add a brief inline comment: `# 8 hex chars = 32-bit ID space; sufficient for single-user session disambiguation`

### Finding C-06
- **File:line**: `orchestration/templates/SESSION_PLAN_TEMPLATE.md:147–165`
- **Severity**: P3
- **Category**: Confusing placeholder — wrong technology
- **Description**: The "Agent Spawn Sequence" section shows Python-style `spawn()` pseudocode with a `background=True` parameter. But the actual spawn mechanism documented in RULES.md is the `Task` tool call (TypeScript/JSON), not Python. A new adopter reading this template will not recognize the pseudocode as pseudocode — it looks like actual implementation syntax, and the `background=True` flag directly contradicts RULES.md's prohibition against `run_in_background`.
- **Suggested fix**: Replace the Python pseudocode with a prose description (e.g., "Spawn via Task tool with `subagent_type='python-pro'`, `model='sonnet'`") or with accurate pseudo-JSON matching the actual tool call structure. Add a comment clarifying this is illustrative, not executable. This is P3 rather than P1 because the template is advisory, but the contradiction with a hard prohibition is worth flagging.

### Finding C-08

- **Description**: The sync-to-claude.sh description says rsync is run "without `--delete`" and "excluding `scripts/` and `_archive/`". ### Finding C-09
- **File:line**: `scripts/sync-to-claude.sh:36`
- **Severity**: P3
- **Category**: Structural over-engineering for a fixed list
- **Description**: The script iterates over a `for script in "$REPO_ROOT/scripts/build-review-prompts.sh"` loop that contains exactly one item. A `for` loop over a single literal is semantically correct but adds cognitive overhead — a reader unfamiliar with the file might expect multiple scripts to be listed. The comment above says "Only build-review-prompts.sh is synced here" but the loop structure implies extensibility that the comment explicitly discourages.
- **Suggested fix**: Replace the loop with a direct copy: `cp "$REPO_ROOT/scripts/build-review-prompts.sh" ~/.claude/orchestration/scripts/build-review-prompts.sh && chmod +x ...`. Keep the missing-file guard as an `if` check before the copy.
