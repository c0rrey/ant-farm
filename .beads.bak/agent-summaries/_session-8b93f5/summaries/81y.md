## 1. Approaches Considered

**Approach A - Legend subheading block after the diagram**
Add a `### Legend` or `#### Acronyms` level-3/4 heading after the closing code fence, followed by a bullet list of four definitions. This is visually distinct and clearly labeled.
- Pro: Unambiguously scannable; heading signals purpose.
- Con: A dedicated heading for four items adds structural weight disproportionate to the content size. Interrupts the flow from diagram to Workflow section.

**Approach B - Inline parenthetical expansions inside the ASCII diagram**
Modify the code fence lines themselves to show `CCO (Colony Cartography Office)` etc. directly inside the box.
- Pro: Definitions are co-located with the acronym's only occurrence.
- Con: The diagram lines are near the 57-character box width. Expanding abbreviations inside would overflow the box borders, breaking ASCII art alignment. High risk of visual corruption.

**Approach C - Single prose sentence after the diagram**
One continuous sentence: "In the diagram above: CCO = Colony Cartography Office; WWD = Wandering Worker Detection; DMVDC = Dirt Moved vs Dirt Claimed; CCB = Colony Census Bureau."
- Pro: Extremely compact; no structural overhead.
- Con: Run-on semicolon lists are harder to scan. The sentence format obscures the key-value structure of the definitions.

**Approach D - Inline bold key-value line, no subheading (SELECTED)**
One line immediately after the closing code fence using `**ACRONYM** = Full Name` separated by pipe characters. No heading required.
- Pro: Scannable (bold terms pop visually), compact (single line), no structural overhead, consistent with README's existing use of bold for emphasis, immediately adjacent to the diagram.
- Con: All four definitions on one line can wrap on narrow viewports, though this is a minor concern for a developer README.
