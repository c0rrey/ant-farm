## Approaches Considered

### 1. Bold style — `**MANDATORY**` everywhere
**Strategy**: Replace all MANDATORY occurrences with bold `**MANDATORY**` at sentence-initial and inline positions. Remove parentheses and dashes from all variants.
**Pros**: Maximum visual salience; bold already used in several places; renders prominently in all markdown viewers.
**Cons**: Awkward in parenthetical contexts like `## Report (MANDATORY)` and `(MANDATORY keyword present)`; forces restructuring of sentences where MANDATORY appears mid-sentence in parens.
**Risks**: `(MANDATORY keyword present)` in checkpoints.md is a literal string checked by a verifier — changing it to `**MANDATORY keyword present**` would desync the check from the template.
**Primary tradeoff**: Visual salience vs syntactic fit in parenthetical contexts.

### 2. Bare parenthetical — `(MANDATORY)` everywhere
**Strategy**: Use `(MANDATORY)` as the single canonical form. Strip bold markers, dashes, and elaborations like `— do not skip` from all variants.
**Pros**: Consistent, minimal, already the form used in dirt-pusher-skeleton and checklist items; integrates cleanly in both step headers and inline positions.
**Cons**: Slightly less emphatic than bold; loses the `— do not skip` reinforcement from step headers.
**Risks**: Low — this form already exists in multiple places and represents the most common variant.
**Primary tradeoff**: Minimal footprint vs maximum emphasis.

### 3. Bold parenthetical — `(**MANDATORY**)` everywhere
**Strategy**: Use `(**MANDATORY**)` in all parenthetical positions; `**MANDATORY**` at document-level warnings.
**Pros**: Bold inside parens is a common convention for important annotations in markdown.
**Cons**: Introduces two rendering forms (bold standalone vs bold inside parens), so not truly single-form. Some markdown renderers strip bold inside parens inconsistently.
**Risks**: `(**MANDATORY**)` is harder to type and search for; inconsistency between sentence-start and parenthetical usage remains.
**Primary tradeoff**: Combines emphasis with parenthetical syntax but at the cost of requiring two forms.

### 4. Context-sensitive style — preserve semantic distinctions
**Strategy**: Use `(MANDATORY)` for step/header annotations; keep `(MANDATORY GATE)` and `(MANDATORY keyword present)` as named concepts; keep `**MANDATORY**` only at document-level warning (L6 of implementation.md).
**Pros**: Respects semantic distinctions already present; named variants carry distinct meaning beyond simple annotation.
**Cons**: Not a single uniform token — technically three forms. Harder to call "standardized" even if justifiable.
**Risks**: Future editors may not understand the distinction and introduce more variation.
**Primary tradeoff**: Semantic fidelity vs true uniformity.

## Selected Approach
**Choice**: Approach 2 — `(MANDATORY)` parenthetical as the canonical style for all keyword annotations. Named semantic variants `(MANDATORY GATE)` and `(MANDATORY keyword present)` preserved unchanged since they carry distinct programmatic meaning.

**Rationale**: The dirt-pusher-skeleton is the canonical template all agent prompts derive from. It already uses `(MANDATORY)` in its step list — making all templates match this established form is the most consistent path. Approach 1 (bold) was rejected because it doesn't integrate naturally into parenthetical contexts already used in reviews.md and checkpoints.md. Approach 3 was rejected because it requires two forms. Approach 4 was rejected because the `**MANDATORY**` at L6 of implementation.md is a document-level notice that does not need to be preserved as a special case — `⚠️` alone provides the visual signal, and `(MANDATORY)` after it keeps formatting consistent.
