## 1. Approaches Considered

### Approach A: Adopt `YYYYMMDD-HHmmss` (big-head-skeleton.md as canonical)
Update all 7 checkpoints.md occurrences and 1 pantry.md occurrence to match the format already defined in big-head-skeleton.md. Zero changes to big-head-skeleton.md since it is already the declared canonical source ("canonical across all orchestration templates").

Tradeoffs:
- Pro: aligns with the file that explicitly declares itself canonical
- Pro: `HHmmss` uses standard datetime notation (uppercase H = hours, lowercase m = minutes, lowercase s = seconds) matching ISO 8601-adjacent convention used in most datetime libraries
- Pro: the UTC qualifier in big-head-skeleton.md is preserved and carried into the checkpoints.md definition line
- Con: requires changing 8 locations across 2 files

### Approach B: Adopt `YYYYMMDD-HHMMSS` (checkpoints.md / pantry.md majority)
Standardize on the all-uppercase format already present in 8 locations, updating only big-head-skeleton.md.

Tradeoffs:
- Pro: fewest total changes (1 file, 1 location)
- Pro: already used in the majority of locations
- Con: all-uppercase `HHMMSS` is technically ambiguous — `MM` conventionally means months in standard datetime notation, not minutes
- Con: ignores the file that explicitly declares itself canonical

### Approach C: Adopt `YYYYMMDD-HHmmss` with UTC qualifier added everywhere
Same as Approach A but also propagate the UTC qualifier to every usage site (not just the definition line in checkpoints.md).

Tradeoffs:
- Pro: most thorough and technically precise
- Con: all-caps UTC annotation in each of the 5 artifact-naming bullet points would be verbose and adds prose beyond what the acceptance criteria require
- Con: slight scope creep — adds information rather than purely standardizing the existing format string

### Approach D: Cross-reference architecture — single definition, all usages defer to it
Rewrite each usage site to say "see Timestamp format above" or link to the canonical definition, eliminating inline format string repetition.

Tradeoffs:
- Pro: single point of truth eliminates any future drift
- Pro: best long-term maintainability
- Con: substantially rewrites template prose across 5 artifact-naming sections — high risk of unintended changes
- Con: changes the information architecture, not just the format string — beyond stated scope

## 2. Selected Approach

**Approach A: Adopt `YYYYMMDD-HHmmss` as the canonical format, sourced from big-head-skeleton.md**

Rationale:
1. `big-head-skeleton.md` line 8 declares its term definitions "canonical across all orchestration templates." This is the intended source of truth; the other files should conform to it, not the reverse.
2. `HHmmss` is semantically correct in standard datetime notation (`H` = hour, `m` = minute, `s` = second). All-uppercase `HHMMSS` risks confusion with `MM` = months.
3. The UTC qualifier on the checkpoints.md definition line (`YYYYMMDD-HHmmss` (UTC)) adds clarity for agents generating filenames without adding verbosity to every usage site.
4. The scope of changes is minimal and contained to the two non-canonical files.
