### Finding 5: Broken anchor link `#workflow-concepts` in RULES.md and checkpoints.md
- **File(s)**: orchestration/RULES.md:37, orchestration/templates/checkpoints.md:237
- **Severity**: P2
- **Category**: correctness
- **Description**: Two files link to GLOSSARY.md using a Markdown anchor:
  - `orchestration/RULES.md:37`: `(see [Glossary: wave](orchestration/GLOSSARY.md#workflow-concepts))`
  - `orchestration/templates/checkpoints.md:237`: `(see [Glossary: wave](orchestration/GLOSSARY.md#workflow-concepts))`

  In GitHub-flavored Markdown (CommonMark), anchors are generated from heading text using the heading slug. The GLOSSARY.md section is titled `## Workflow Concepts` (line 40). The slug for this heading would be `#workflow-concepts` — which does match. However, the section heading in the file is `## Workflow Concepts` (capital W and C), and GitHub Markdown anchor generation lowercases and hyphenates, producing `#workflow-concepts`. So the anchor itself is correct for GitHub rendering. **However**, both links use `orchestration/GLOSSARY.md#workflow-concepts` as the path. This is a repo-root-relative path. When checkpoints.md is read at runtime from `~/.claude/orchestration/templates/checkpoints.md`, the relative path `orchestration/GLOSSARY.md` would resolve incorrectly (to a non-existent path). For RULES.md (which also references the path as `orchestration/GLOSSARY.md`), the same issue exists. The links work correctly only in a GitHub-rendered view of the repo root, not at runtime.

  **Severity clarification**: Since RULES.md and checkpoints.md are documentation/instruction files for AI agents (not code that executes links), and agents navigate by file reads not link clicks, this is an operational correctness concern if agents were ever expected to follow these links. For human readers browsing the repo on GitHub, the link works. Downgrading to P2 because it is a navigational aid, not a data flow issue.
- **Suggested fix**: Since the files note the Path Reference Convention (all paths are repo-root relative), the links are internally consistent with that convention. The issue is minor; however, if runtime correctness is critical, add a comment noting the runtime path translation: `(see Glossary: wave — runtime path: ~/.claude/orchestration/GLOSSARY.md#workflow-concepts)`.
- **Cross-reference**: The `orchestration/RULES.md` Path Reference Convention explicitly states "all file paths use repo-root relative format" — so these links follow the stated convention. This finding may be a known-acceptable trade-off.

### Finding 8: Big Head timeout escalation path is architecturally unexecutable inside TeamCreate (cross-review flag from edge-cases-reviewer)
- **File(s)**: orchestration/templates/reviews.md:657-665, orchestration/templates/big-head-skeleton.md:89,92
- **Severity**: P1
- **Category**: correctness
- **Description**: The Pest Control timeout path in reviews.md Step 4 and big-head-skeleton.md step 9 instructs Big Head to "escalate to the Queen immediately" after 120 seconds of no response. However, Big Head runs inside a TeamCreate team and has no channel to reach the Queen during team execution. Inside a team, agents can only: (1) SendMessage to named team members, (2) write files to disk, or (3) return output when the team terminates.

  The escalation block (reviews.md:659-664) is formatted as a message body but has no specified recipient and no SendMessage call. There is no mechanism for this message to reach the Queen while the team is running. RULES.md Step 3b confirms: "After team completes, DMVDC and CCB have already run inside the team" — the Queen's next action is Step 3c: "Read the consolidated review summary." The Queen reads a file post-hoc; it has no in-band signal channel from the running team.

  Additionally, the protocol does not instruct Big Head to write a fallback artifact to disk in the timeout case. If Big Head hits the timeout and stops without writing the consolidated report (because it cannot file beads and has no other path), the Queen will read a missing or incomplete `review-consolidated-*.md` with no explanation for the failure. The system enters a silent dead-end state.

  The analogous "FAIL" escalation path (reviews.md:668-677) has the same structural problem: it also says "Big Head MUST escalate to the Queen" with no mechanism specified.
- **Suggested fix**: Replace "escalate to the Queen immediately" with "write an error artifact to disk." Big Head should write a `review-consolidated-BLOCKED-<timestamp>.md` (or append an escalation section to the normal output path) describing the timeout condition, then terminate. The Queen reads the session artifacts post-team-completion and will encounter this file. RULES.md Step 3c should be updated to check for `*-BLOCKED-*` artifacts before reading the consolidated summary, and surface them to the user. This preserves the TeamCreate architecture while giving the Queen an observable signal.
- **Cross-reference**: Flagged by edge-cases-reviewer. Structurally related to how the Big Head "FAIL" escalation path (reviews.md:668-677) also specifies an unexecutable Queen escalation. Same root cause — the protocol was written assuming Big Head has a real-time channel to the Queen, but TeamCreate does not provide one.

### Finding 7: README.md Step 6 code block instructs deletion of session artifacts — contradicts retention policy (cross-review flag from clarity-reviewer)
- **File(s)**: README.md:218
- **Severity**: P1
- **Category**: correctness
- **Description**: README.md Step 6 "Land" contains a bash code block with `rm -rf .beads/agent-summaries/_session-*/`. This directly contradicts the documented session artifact retention policy established in commit 8f24d54 and codified in both `CLAUDE.md:72` and `~/.claude/CLAUDE.md:70`: "Session artifacts in `.beads/agent-summaries/_session-*/` are kept for posterity — do NOT delete them during cleanup."

  README.md is the primary onboarding and reference document. Any user following the Step 6 code block verbatim will delete all session history — which the policy explicitly prohibits. RULES.md:133 (the Queen's actual runtime instruction file) correctly omits this deletion command: its Step 6 says only "git pull --rebase, bd sync, git push, clean up stashes and remote branches" — no `rm -rf`. The README is the sole source of this erroneous instruction.

  This is P1 because it causes data loss (permanent deletion of session audit history) for any user following the documented procedure. The CLAUDE.md policy was established specifically to prevent this, and the README directly undermines it.
- **Suggested fix**: Remove `rm -rf .beads/agent-summaries/_session-*/` from README.md:218. Optionally replace with a comment: `# Session artifacts retained for posterity — prune manually if needed (see CLAUDE.md)`.
- **Cross-reference**: Flagged by clarity-reviewer. RULES.md:133 is the correct reference — it does NOT include the deletion command. CLAUDE.md:72 is the authoritative policy source.
