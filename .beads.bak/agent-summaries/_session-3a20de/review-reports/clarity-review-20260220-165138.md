## Findings Catalog

### Finding 1: Progress log entry references non-existent `big-head-summary.md` filename
- **File(s)**: `orchestration/RULES.md:186`
- **Severity**: P2
- **Category**: clarity
- **Description**: The progress log echo in Step 3b-v uses the filename `big-head-summary.md` as the report path, but the actual consolidated output is written to `review-consolidated-{timestamp}.md` (as defined throughout reviews.md, checkpoints.md, and pantry.md). A recovery Queen reading this log entry would be directed to a file that does not exist. This is a misleading fact embedded in an instruction rather than a stale comment — it has operational consequences.
  ```
  echo "...|report=${SESSION_DIR}/review-reports/big-head-summary.md" >> ${SESSION_DIR}/progress.log
  ```
- **Suggested fix**: Change `big-head-summary.md` to `review-consolidated-${TIMESTAMP}.md` (or to a glob pattern note if the timestamp is not available at that point), matching the actual output path convention used everywhere else.
- **Cross-reference**: Correctness reviewer should also examine — the mismatch could cause a crash-recovery Queen to read the wrong file.

### Finding 2: `{TIMESTAMP}` variable used before it is defined in the dummy reviewer send-keys command
- **File(s)**: `orchestration/RULES.md:176`
- **Severity**: P2
- **Category**: clarity
- **Description**: The `tmux send-keys` command in Step 3b-v interpolates `${TIMESTAMP}` as a shell variable:
  ```bash
  "Write your report to ${SESSION_DIR}/review-reports/dummy-review-${TIMESTAMP}.md"
  ```
  However, the step narrative only states that the Queen generates a timestamp "at the start of Step 3b" (Step 3b-i). There is no instruction in Step 3b-v to bind that timestamp to a variable named `TIMESTAMP`, nor is the variable set anywhere in the surrounding code block. A reader following these instructions literally would produce a command with an empty or undefined `${TIMESTAMP}`. The instruction is ambiguous: is the Queen supposed to have already set `TIMESTAMP` earlier, or is this a new variable to be set here?
- **Suggested fix**: Add an explicit variable assignment before the `tmux send-keys` command, e.g., `TIMESTAMP=$(date +%Y%m%d-%H%M%S)`, or add a note cross-referencing Step 3b-i: "Use the `TIMESTAMP` variable generated in Step 3b-i."
- **Cross-reference**: Edge Cases reviewer — if `TIMESTAMP` is unset, the shell substitution produces an empty string, writing the report to `.md`, which is a real file-creation edge case.

- **File(s)**: `scripts/parse-progress-log.sh:61-62`
- **Severity**: P3
- **Category**: clarity
- **Description**: The comment block at line 61 uses "step_key" (with underscore) in the description of each entry's fields ("Each entry: 'step_key|display_label|description'") but the preceding comment header at line 61 says "Ordered step definitions: step_key | display_label | description". Both are consistent with the code, but the comment at line 108 ("Format: TIMESTAMP|step_key|field=value|...") introduces a slight inconsistency: the actual log format written by RULES.md uses plain step names like `step0`, `step1` without any framing terminology. The mismatch between the "step_key" label in comments and the plain "step0"-style values in the actual log is minor but could confuse a new contributor.

### Finding 5: `parse-progress-log.sh` — "floor of 3 applies" table note is inaccurate
- **File(s)**: `orchestration/templates/checkpoints.md:431`
- **Severity**: P3
- **Category**: clarity
- **Description**: The worked examples table in the DMVDC Check 1 section contains the note "3 (floor of 3 applies)" for N=6. The formula is `max(3, min(5, ceil(N/3)))`. For N=6: `ceil(6/3)=2`, `min(5,2)=2`, `max(3,2)=3`. The `max(3,...)` is a **floor** on the result (it imposes a minimum of 3), not a "floor" in the standard mathematical sense (which would round down). Calling it "floor of 3" may confuse readers who expect "floor" to mean rounding down. A reader who interprets "floor" as the math floor function (`⌊x⌋`) will be confused.
- **Suggested fix**: Change the note to "3 (minimum of 3 applies)" or "3 (max(3,...) enforces minimum)".
- **Cross-reference**: None.
