## 1. Approaches Considered

### Approach A: Inline RULES.md detection prose only (no new script)
Add plain English instructions directly into Steps 0 and the Session Directory section of RULES.md telling the Queen to check for an existing `progress.log`, parse it manually using inline commands, and present a formatted resume plan.

- Pros: Zero new files; simpler surface area.
- Cons: The Queen must interpret raw pipe-delimited log lines and build a structured plan in its context window. Error-prone, hard to test, and produces inconsistent output depending on context window state. Parsing logic in prose is ambiguous.

### Approach B: New shell script writing a resume-plan.md artifact (SELECTED)
Create `scripts/parse-progress-log.sh` that reads progress.log, determines completed/in-progress/pending steps for all 9 workflow milestones, writes a structured `resume-plan.md` to the session directory, and exits with a meaningful exit code. RULES.md Step 0 instructs the Queen to run the script and present the plan verbatim.

- Pros: Clean separation of concerns. The Queen reads one pre-formatted markdown file. Consistent with existing script patterns (`fill-review-slots.sh`, `compose-review-skeletons.sh`). Exit codes make branching logic unambiguous. Independently testable.
- Cons: Adds a bash 4+ dependency for associative arrays (shared with existing scripts).

### Approach C: Script outputs to stdout only (no file written)
Script parses progress.log and prints the resume plan to stdout. RULES.md tells the Queen to capture and present stdout.

- Pros: Slightly simpler script.
- Cons: No artifact file for debugging after the fact. Inconsistent with existing script pattern where scripts write files and return exit codes. Stdout capture in the Queen's context is fragile.

### Approach D: Python script
Same logic as Approach B but implemented in Python for richer data manipulation.

- Pros: More expressive parsing.
- Cons: Adds a Python runtime dependency. All existing scripts are bash. Unnecessary complexity for a log-parsing task where awk/bash are sufficient. Inconsistent with repo conventions.

---

## 2. Selected Approach with Rationale

**Approach B** was selected. It follows the established script pattern in this repo (file-based output, exit codes for branching, `set -euo pipefail`, `#!/usr/bin/env bash`). The Queen's job is to run one script and read one markdown file — no log parsing in context. The three-exit-code contract (0 = resume available, 1 = error, 2 = session complete) maps cleanly to the three RULES.md branches.

## 3. Implementation Description

### Files Changed

**`scripts/parse-progress-log.sh`** (new, 231 lines)

A bash script that:
1. Validates the `SESSION_DIR` argument and confirms `progress.log` exists and is readable.
2. Defines all 9 workflow step keys (`step0` through `step6`, including `step3b` and `step3c`) in order, with `step_label()` and `step_resume_action()` helper functions.
3. Reads `progress.log` line by line, splitting on `|` to extract `timestamp`, `step_key`, and `rest`. Populates associative arrays `STEP_COMPLETED`, `STEP_TIMESTAMP`, `STEP_DETAILS`.
4. Checks for `step6` completion: exits with code 2 if the session already completed.
5. Identifies the first uncompleted step as the resume point.
6. Writes `{SESSION_DIR}/resume-plan.md` — a markdown document containing: a status table (COMPLETE with timestamp / RESUME HERE / pending) for all steps, a "Resume Action" section with the concrete next action, and a "Last Completed Milestone" section with logged details.
7. Exits with code 0 on success, 1 on any error.

**`orchestration/RULES.md`** (modified, two sections)

- **Step 0 (lines 55-72)**: Added a "Crash recovery detection" block after the existing Step 0 body. Instructs the Queen to check for a prior SESSION_DIR before generating a new SESSION_ID, run `parse-progress-log.sh`, and await user reply (`resume` or `fresh start`) before taking any action.
- **Session Directory section (lines 295-309)**: Added `resume-plan.md` to the artifacts list with its description, and added a "Crash recovery script" reference block documenting the three exit codes.

## 4. Correctness Review

### scripts/parse-progress-log.sh

- Header/usage/exit codes documented accurately and consistently with script behavior.
- `set -euo pipefail` present; matches all existing scripts.
- Argument validation: checks `$#`, directory existence, file existence, file readability — all validated before any processing.
- Step key order (`step0..step6` including `step3b`, `step3c`) matches the progress log `echo` commands in RULES.md exactly.
- `step_label()` and `step_resume_action()` cover all 9 keys plus a `*)` fallback.
- Log parsing: `IFS='|' read -r timestamp step_key rest` correctly splits the pipe-delimited format.
- Blank/malformed line guard: `[ -z "$step_key" ] && continue` prevents empty keys from polluting associative arrays.
- Multi-occurrence steps (step2, step3, step3b, step3c): last occurrence wins for `STEP_TIMESTAMP`/`STEP_DETAILS`, and `STEP_COMPLETED` is set on first occurrence. Correct: for the resume plan, what matters is that the step has been logged at all.
- Exit 2 path: checked before resume-point determination, so a complete session never writes a resume-plan.md.
- Resume point: iterates `STEP_KEYS` in order, stops at first absent key. The fallback (`RESUME_STEP="step6"`) handles the edge case where all steps 0-5 are logged but step6 is absent.
- Output table: each step gets COMPLETE (with timestamp), RESUME HERE, or pending — exhaustive and exclusive.
- `LAST_COMPLETED`: correctly iterated in order; final assignment is the highest-completed step.
- `${STEP_DETAILS[$LAST_COMPLETED]:-}` safely handles empty `rest` field without triggering `set -u` error.
- Output file write: uses brace-grouped heredoc redirected to `$OUT_FILE`; error check present.

No test framework exists in this repo for orchestration scripts. Manual validation approach:

1. Create a test session directory with a partial `progress.log` (e.g., only step0 and step1 logged).
2. Run `bash scripts/parse-progress-log.sh .beads/agent-summaries/_session-test`.
