## 1. Approaches Considered

### Approach A: Footnote row or text below the table
Add a text footnote immediately below the table (outside the table structure): "Note: Each CCB re-run counts as 1 toward both the per-checkpoint limit above and the session total."

Tradeoffs:
- Keeps the table schema unchanged
- Easy to spot when reading the table
- Footnote-style placement can be overlooked if the reader stops after the table rows

### Approach B: Add a fourth "Notes" column to the table
Add a `Notes` column to the table and populate the CCB row with the counter interaction text. All other rows get an empty notes cell.

Tradeoffs:
- Information is co-located with the CCB row — maximum discoverability
- Changes the table schema, requiring empty cells in three other rows
- Adds visual noise; inconsistent cell density
- Makes the CCB row stand out in a way that could imply other rows have no interaction effects

### Approach C: Clarifying sentence immediately after the table, before "Track retry count" (SELECTED)
Insert one sentence between the table and the existing "Track retry count" sentence, naming both limit values explicitly: "Counter interaction: each CCB re-run counts as 1 toward both the per-checkpoint limit (1) and the session total (5)."

Tradeoffs:
- Minimal change — no table restructuring
- Natural reading flow: reader finishes the table and immediately encounters the clarification
- Both numeric limits (1 and 5) are named explicitly, satisfying the acceptance criterion without ambiguity
- Adding a second sentence about what happens when the per-checkpoint limit is hit adds completeness

### Approach D: Embed counter interaction in the CCB row's "After Limit" cell
Change the "CCB fails" row's After Limit cell to: "Present to user with verification report attached (each re-run counts toward both this limit and the session total)."

Tradeoffs:
- Information is inline in the table
- Makes the cell verbose and stylistically inconsistent with other rows
- Parenthetical in a table cell is harder to scan
- Does not make the numeric values of both limits explicit

## 2. Selected Approach with Rationale

**Selected: Approach C.**

The acceptance criterion requires the table to "explicitly state" how per-checkpoint retries interact with the session total. A sentence immediately after the table, naming both limit values (1 and 5) and identifying the specific scenario (CCB re-run), satisfies this precisely. It is the least invasive change — no table schema changes, no new sections — and it sits in the natural reading flow after the table. A second sentence was added to complete the interaction description: what happens when the per-checkpoint limit is hit (it still counts against the session total).
