**Task**: Add append-only progress log for Queen crash recovery
**Status**: complete

## 1. Approaches Considered

### Approach A: Inline append commands embedded in each step's prose
Add the `echo ... >> progress.log` shell command directly into each step's narrative description text as a sentence. The command becomes part of the flowing prose.

Tradeoffs:
- Maximally explicit — the command is right where the step is described
- Hard to visually distinguish from surrounding operational instructions
- Increases prose density, making steps harder to scan for their primary action

### Approach B: Separate labeled subsection after each step block
After each step's main description, insert a clearly labeled subsection heading (e.g., `#### Progress Log Entry`) with the exact append command below it.

Tradeoffs:
- Very readable and easy to locate or skip
- Adds 3 lines per step (heading + blank + command), significantly increasing total line count
- Section headings at h4 depth may be visually overstructured for what is a one-liner

### Approach C: Single consolidated "Progress Log" section listing all milestones
Add one new top-level section to RULES.md listing all milestones, their fields, and the command template. Reference it from the Session Directory section. Do not touch the step descriptions.

Tradeoffs:
- Zero disruption to the step descriptions (minimal diff)
- The Queen must cross-reference two sections to understand what to do at each step
- Recovery logic is in one place (good for readers looking for the format), but disconnected from the workflow steps
- Higher cognitive load during execution: "What do I write here? Let me go look it up."

### Approach D: Inline callout lines at the end of each step, plus a path-definition entry in Session Directory (SELECTED)
Append a compact `**Progress log:**` line at the end of each step showing the exact fields for that step's entry, using `>>` (append-only) redirection. Also add one bullet to the Session Directory section naming `progress.log` and its purpose.

Tradeoffs:
- Context stays close to the action — highest usability for the Queen during execution
- Visual weight per step is low: one line with a bold label makes it easy to scan past if not in recovery mode
- The append-only contract is explicit at every callout point (`>>` operator)
- Recovery sessions find the log format in Session Directory and the per-step fields inline
- No new top-level section required

## 2. Selected Approach with Rationale

**Selected: Approach D.**

The task requires the Queen to append one line per workflow milestone. Inline callouts place the instruction exactly where the action occurs, which is the highest-reliability placement for an operational rule — the Queen reads the step and immediately sees what to log. Approach C (separate section) risks the Queen omitting the log entry because the instruction is not co-located with the step. Approach B's subsection headings add unnecessary structural weight. Approach A's inline prose is harder to visually distinguish from the step's main instruction.

The Session Directory bullet satisfies AC4 (path documentation) and adds the append-only/recovery-purpose description without requiring a new section.
