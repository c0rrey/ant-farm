### Finding 7: compose-review-skeletons.sh — `extract_agent_section` function comment says "exactly one such delimiter" but YAML frontmatter in the skeleton files also uses `---`

- **File(s)**: `scripts/compose-review-skeletons.sh:69-76`
- **Severity**: P3
- **Category**: clarity
- **Description**: The `extract_agent_section` helper at lines 69-76 is documented as: "Skeleton template files are expected to contain exactly one such delimiter separating the instruction block from the agent-facing content." However, markdown files with YAML frontmatter use `---` as the opening and closing delimiter of the frontmatter block. If a skeleton file ever has frontmatter (two `---` lines), `awk` with `count>=1` would start printing after the first `---` (the frontmatter opener) rather than the intended content separator. The current skeleton files may not have frontmatter, but the comment's assertion ("exactly one such delimiter") does not warn the reader about this limitation.
- **Suggested fix**: Clarify the comment: "ASSUMPTION: skeleton files have no YAML frontmatter — only a single `---` separator between the instruction block and agent-facing content. Files with YAML frontmatter (two `---` lines) will not be handled correctly."

### Finding 8: compose-review-skeletons.sh — `sed` regex comment uses "2+ chars, all-caps with underscores" but the pattern also matches 1-char uppercase words

- **File(s)**: `scripts/compose-review-skeletons.sh:103-108` and `scripts/compose-review-skeletons.sh:157-163`
- **Severity**: P3
- **Category**: clarity
- **Description**: The comment at line 103 reads: "Pattern: {WORD} → {{WORD}} where WORD matches [A-Z][A-Z_]* (2+ chars, all-caps with underscores). Single-char tokens like {X} do NOT match and are left unchanged." However, the regex `[A-Z][A-Z_]*` matches any uppercase letter followed by zero or more uppercase-or-underscore characters. This means a single uppercase letter `A` preceded by `{` would match as `{A}` → `{{A}}`. The pattern actually requires at least 1 uppercase char (minimum: `{A}`), not 2. The comment says "2+" which is wrong — `[A-Z][A-Z_]*` is "1+". The intended behavior (only converting `{WORD}` tokens of 2+ chars) would require `[A-Z][A-Z_]+` (note the `+`). This comment/regex mismatch could mislead a future maintainer about what the pattern actually matches.
- **Suggested fix**: Either fix the regex to `[A-Z][A-Z_]+` (enforcing 2+ chars as the comment states), or update the comment to accurately state "1+ uppercase chars" and provide an example of why `{A}` is considered harmless in practice. The same issue exists identically at line 163 — fix both.

### Finding 9: parse-progress-log.sh — POSIX-compatibility comment contradicts bash-ism in the same block

- **File(s)**: `scripts/parse-progress-log.sh:104-150`
- **Severity**: P3
- **Category**: clarity
- **Description**: The map store section (lines 104-150) is headed "POSIX-compatible key-value store using a temp directory. Replaces bash 4+ declare -A associative arrays." However, the script uses `[[ "$timestamp" =~ regex ]]` (line 169), which is a bash-ism not available in POSIX sh. The shebang is `#!/usr/bin/env bash` (line 1), so the script is bash-only. The POSIX compatibility rationale in the comment is therefore narrowly scoped (only to the map store, not the whole script), but the heading "POSIX-compatible" reads as a global property claim, which is misleading. The actual motivation was to avoid `declare -A` which requires bash 4+, not to write POSIX sh.
- **Suggested fix**: Reword the comment heading to: "Bash 3+-compatible key-value store (replaces bash 4+ `declare -A`). Uses a temp directory to avoid associative arrays." This makes the scope clear without implying POSIX sh compatibility.

### Finding 10: parse-progress-log.sh — UNREACHABLE comment is technically accurate but the reasoning it provides is incomplete

- **File(s)**: `scripts/parse-progress-log.sh:206-211`
- **Severity**: P3
- **Category**: clarity
- **Description**: The UNREACHABLE comment at lines 206-211 reads: "RESUME_STEP is always set by the loop above because SESSION_COMPLETE is in STEP_KEYS. If SESSION_COMPLETE were completed, the early-exit guard above (exit 2) would have already terminated the script." This reasoning is correct, but only for the case where `SESSION_COMPLETE` has been logged. A second path to unreachability is: the for loop over `STEP_KEYS` iterates until a key is not completed, which means it always sets `RESUME_STEP` to some key unless ALL keys are completed — but the early-exit guard handles the ALL-completed case. The comment only explains one branch of the proof. While the comment is not wrong, it may leave future maintainers uncertain about the case where no step at all has been logged (all steps pending) — in that case, the loop sets `RESUME_STEP = SESSION_INIT` on the first iteration.
- **Suggested fix**: Expand the comment: "UNREACHABLE: The loop always sets RESUME_STEP on its first iteration (SESSION_INIT is never completed if we reach here — the SESSION_COMPLETE guard above handles the only path where all steps are logged). This branch is retained as a defensive assertion."

## Preliminary Groupings

### Group A: Duplicate/redundant content creating ambiguity about the canonical version

- Finding 1 (SETUP.md Quick Setup vs Recipe Card duplication)
- Finding 2 (SETUP.md Step 1 numbering collision between sections)
- **Root cause**: Two sections in SETUP.md serving overlapping purposes — a reference snippet and a prose walkthrough — were added separately without reconciling shared content.
- **Suggested combined fix**: Designate one section as canonical (Recipe Card for copy-paste, Quick Setup for prose walkthrough) and have the other section reference it rather than repeat it.

### Group B: Misleading/inaccurate or incomplete comments in bash scripts

- Finding 8 (compose-review-skeletons.sh: `sed` pattern comment says "2+" but regex matches "1+")
- Finding 9 (parse-progress-log.sh: "POSIX-compatible" heading overstates scope of portability claim)
- Finding 13 (compose-review-skeletons.sh: header comment at line 116 omits `{{TASK_IDS}}` from its slot list)
- **Root cause**: Comments describing regex/language constraints or slot contracts were not kept in sync with the actual implementation as new slots or patterns were added.
- **Suggested combined fix**: Audit all bash script comments that make specific quantitative, compatibility, or enumeration claims and verify them against the actual code.

### Group C: Insufficient inline documentation for non-obvious design choices

- Finding 5 (reviews.md: polling loop uses intentional placeholder strings with no explanation)
- Finding 7 (compose-review-skeletons.sh: `extract_agent_section` comment doesn't warn about YAML frontmatter assumption)
- Finding 10 (parse-progress-log.sh: UNREACHABLE comment reasoning incomplete)
- **Root cause**: Comments were added to explain what the code does but not why certain unusual patterns (placeholder-as-test-input, single-delimiter assumption, unreachability proof) exist.
- **Suggested combined fix**: For each non-obvious pattern, add a "WHY" sentence to the adjacent comment.

### Group D: UI/navigation issues in markdown documents

- Finding 3 (pantry.md: deprecated section not visually prominent enough)
- Finding 4 (pantry.md: inconsistent imperative phrasing across condition blocks)
- Finding 6 (reviews.md: Round-Aware Protocol section placed after sections that reference it)
- Finding 11 (README.md: deprecated agent row inconsistently struck through)
- Finding 12 (pantry-review.md archived: past/present tense inconsistency)
- **Root cause**: As documents evolved, sections were added or deprecated without a full read-through to ensure consistency of tone, ordering, and visual signaling.
