# ant-farm Glossary

Canonical definitions for all framework terms, checkpoint acronyms, and ant metaphor role names used across the orchestration system. This document is the single source of truth — when a term appears in `RULES.md`, `checkpoints/`, `reviews.md`, or any template, its meaning is defined here.

---

## Workflow Concepts

| Term | Definition |
|------|------------|
| **session** | A single invocation of the orchestration system, from "let's get to work" through `git push`. Identified by a unique session ID and backed by a session directory (`.crumbs/sessions/_session-{id}/`). |
| **wave** | A batch of implementation agents that run concurrently within a session. Wave boundaries are chosen to avoid file conflicts: tasks that touch the same file are placed in different waves. Wave N+1 does not start until all agents in wave N have committed and passed scope-verify. |
| **checkpoint** | A mandatory verification gate that blocks the next phase of work until it returns PASS (or an approved WARN). There are six checkpoints: startup-check, pre-spawn-check, scope-verify, claims-vs-code, review-integrity, and session-complete. |
| **scope boundary** | The explicit list of files and line ranges an agent is permitted to edit. Defined in the agent's data file. Changes outside the scope boundary are a scope-verify violation. |
| **data file** | A per-task artifact written by the Pantry that contains the task's scope boundaries, affected files, root cause, acceptance criteria, agent type, and off-limits areas. The agent reads this file at Step 0. |
| **briefing** | A ~40-line summary written by the Scout to `{session-dir}/briefing.md`. Contains ready/blocked task counts, proposed execution strategies with wave groupings, risk assessments, and agent type recommendations. The Queen reads only this — not raw task data. |
| **preview file** | A combined file written by the Pantry to `{session-dir}/previews/` that merges the agent prompt skeleton with the task's data file. The Checkpoint Auditor audits these under pre-spawn-check before the Queen spawns any agent. |
| **verdict** | The outcome of a checkpoint audit. Valid values: PASS, WARN (pre-spawn-check and scope-verify only), PARTIAL (claims-vs-code and review-integrity only), or FAIL. See `checkpoints/common.md` for per-checkpoint thresholds. |
| **information diet** | The design constraint that the Queen never reads source code, tests, configs, or implementation templates. It reads only the Scout's briefing, agent notifications, commit messages, and verdict tables. |
| **escalation** | When a failing agent or stuck checkpoint exceeds its retry limit and the Queen surfaces the problem to the user with full context rather than retrying again. |
| **adjacent issue** | A defect or improvement opportunity noticed by an agent that falls outside its scope boundary. Agents document these in their summary docs but do not fix them. |
| **summary doc** | A structured artifact written by each agent to `{session-dir}/summaries/{task-suffix}.md` at Step 6. Required sections: approaches considered, selected approach, implementation description, correctness review, build/test validation, and acceptance criteria checklist. Note: `{session-dir}` resolves to a path under `.crumbs/sessions/`. |
| **hard gate** | A checkpoint that must return PASS before the system proceeds to the next phase. All six checkpoints (startup-check, pre-spawn-check, scope-verify, claims-vs-code, review-integrity, session-complete) are hard gates. |
| **context window** | The token budget available to a model in a single session. The Queen's information diet keeps its context window clean by offloading reads to subagents. |
| **setup script** | `scripts/setup.sh` — the distribution mechanism for ant-farm. Installs agent definitions, orchestration files, skills, and the `crumb` CLI to their runtime locations (`~/.claude/` and `~/.local/bin/`). For CLAUDE.md: removes any existing ant-farm sentinel block from the global `~/.claude/CLAUDE.md` and the prompt-dir (migration cleanup), then writes the orchestration block from `orchestration/templates/claude-block.md` to the repo's `CLAUDE.md`. Backs up existing files with timestamped `.bak` suffixes before modifying. Idempotent: re-running updates changed files. |
| **Scribe** | An `ant-farm-session-scribe` agent (model: sonnet) spawned by the Queen at Step 5b. The Scribe reads all session artifacts — briefing.md, summaries/*.md, review-consolidated-*.md, and progress.log — then runs git diff/log over the session's commit range. It produces two outputs: (1) `{SESSION_DIR}/exec-summary.md`, the canonical session record, and (2) a prepended CHANGELOG entry in `CHANGELOG.md`. The Scribe is the only agent that writes CHANGELOG.md; the Queen commits that file at Step 6. |
| **exec summary** | A structured session artifact written by the Scribe to `{SESSION_DIR}/exec-summary.md`. Contains five required sections: At a Glance (summary table), Work Completed (per-task outcomes), Review Findings (Nitpicker/Big Head results), Open Issues (crumbs that remain open after the session), and Observations (narrative notes). The CHANGELOG entry is derived from the exec summary — every task ID and commit in the exec summary must also appear in the CHANGELOG. |
| **ESV** | Exec Summary Verification (historical acronym; checkpoint file is now `session-complete.md`). A Checkpoint Auditor (`ant-farm-checkpoint-auditor`, model: haiku) run at Step 5c that must PASS before the Queen proceeds to Step 6 (push). session-complete verifies six things: task coverage (every session task ID appears in the exec summary), commit coverage (every commit hash is accounted for), open crumb accuracy (listed crumbs are actually open), CHANGELOG derivation fidelity (exec summary content matches CHANGELOG), section completeness (all five required exec summary sections present), and metric consistency (counts in the "At a Glance" table match body section counts). Artifact written to `{SESSION_DIR}/pc/pc-session-session-complete-{timestamp}.md`. On session-complete FAIL, the Queen re-spawns the Scribe with the specific violations (max 1 retry); on a second FAIL, the Queen escalates to the user. |
| **lite mode** | A single-crumb execution path defined in `orchestration/RULES-lite.md`. Omits Scout, startup-check, Pantry, Nitpicker review team, review-integrity, session-complete, and the Scribe. Preserves pre-spawn-check (CCO) and claims-vs-code (CMVCC) as hard gates. The implementer reads task files directly (no Pantry pre-digestion). Includes a mandatory self-review step where the implementer verifies its own changes against the crumb's acceptance criteria before committing. Progress log entries include `mode=lite` as a parseable metadata field. Use lite mode only for single, isolated tasks with well-specified acceptance criteria. |
| **self-review** | A mandatory step in lite mode (Step 4 of the implementer's 6 mandatory steps) where the implementer re-reads every changed file and verifies each acceptance criterion is genuinely met. Documents review in the summary doc with file-specific notes — generic "looks clean" language fails claims-vs-code Check 4. Self-review replaces the Nitpicker review team in lite mode but does not replace the claims-vs-code (CMVCC) checkpoint. |
| **PRD** | Product Requirements Document. A Markdown file provided by the user as input to `/ant-farm-plan`. PRDs are the third supported input type alongside freeform requests and structured specs. A PRD may be well-structured (Class A: contains numbered requirements and testable ACs) or narrative (Class B: prose requirements with no testable ACs). Class A PRDs are processed by the PRD Importer; Class B PRDs trigger Surveyor fallback. |
| **PRD Importer** | A subagent spawned by the Planner at Step 1 when input is classified as a PRD file path. Uses `orchestration/templates/prd-import.md` as its instruction template. The PRD Importer validates the PRD file, classifies it (Class A/B/C), extracts requirements into REQ-N / AC-N.M format, presents the extraction to the user for confirmation, and writes `{DECOMPOSE_DIR}/spec.md` in the same format as the Surveyor's output. On Class B or C input, returns a fallback or error signal rather than writing spec.md. Uses `ant-farm-spec-writer` as its subagent type and requires `model: "opus"`. |
| **PRD classification** | The outcome of the PRD Importer's Step 2 analysis. Three classes: **Class A** (well-structured — has numbered requirements and testable ACs; processed to spec.md), **Class B** (narrative — has prose requirements but no testable ACs; triggers Surveyor fallback), **Class C** (invalid — no requirement signals found; triggers error). Recorded in the progress log as `PRD_CLASSIFY|class=<A\|B\|C>`. |
| **extraction diagnostic** | A count summary produced by the PRD Importer after Step 3 (extraction). Reports: paragraphs/headings scanned, requirements extracted, ACs extracted, and ACs flagged as vague. Shown to the user in the confirmation step so they can spot truncation or misclassification. Also recorded verbatim in the `## Assumptions` section of the generated spec.md. |
| **Surveyor fallback** | The path taken when a PRD Importer returns `fallback_to_surveyor`. The Planner spawns the Surveyor (Step 2) with the PRD file content included as `{FEATURE_REQUEST}` context, and tells the user why the fallback happened before spawning. The Surveyor then asks clarifying questions and produces spec.md through its normal workflow. Fallback is not a failure — it is the expected path for narrative PRDs. |

## Checkpoint Names

All six checkpoints are executed by the Checkpoint Auditor (`ant-farm-checkpoint-auditor`). Full definitions live in `orchestration/templates/checkpoints/` (one file per checkpoint type, plus `common.md` shared preamble).

| Checkpoint file | Historical acronym | When it runs | What it verifies | Blocks |
|-----------------|--------------------|--------------|------------------|--------|
| **startup-check.md** | SSV (Scout Strategy Verification) | After Scout returns briefing, before Pantry is spawned | Strategy mechanical correctness: wave groupings are valid, file conflict analysis is accurate, task-to-wave assignments match the file modification matrix, no intra-wave dependency violations | Pantry spawn |
| **pre-spawn-check.md** | CCO (Colony Cartography Office) | After Pantry composes prompts, before agent spawn | Pre-spawn prompt quality: real task IDs, real file paths with line numbers, root cause text, all 6 mandatory steps, scope boundaries, commit instructions, and line-number specificity. Runs in two configurations: impl (reviewing implementation prompts) and review (reviewing Nitpicker/Big Head prompts in Step 3b) | Agent spawn |
| **scope-verify.md** | WWD (Wandering Worker Detection) | After agent commits, before next agent in same wave spawns | Post-commit scope verification: files changed in the commit match the task's expected scope — no scope creep | Next agent in wave |
| **claims-vs-code.md** | CMVCC (Crumbs Moved vs Crumbs Claimed) | After agent completes its summary doc | Substance verification: git diff matches summary claims, acceptance criteria are genuinely met, 4 design approaches are substantively distinct, correctness review is specific rather than boilerplate | Task closure |
| **review-integrity.md** | CCB (Colony Census Bureau) | After the Nitpicker team and Big Head complete, before results are presented to the user | Consolidation integrity: finding counts reconcile, every filed issue has required fields, priority calibration is correct, traceability matrix is complete, deduplication is accurate, no unauthorized issues were filed | Presenting results to user |
| **session-complete.md** | ESV (Exec Summary Verification) | After Scribe writes exec-summary.md and CHANGELOG entry, before Step 6 (push) | Exec summary integrity: task coverage, commit coverage, open crumb accuracy, CHANGELOG derivation fidelity, section completeness, and metric consistency | Step 6 (push) |

## Ant Metaphor Roles

| Name | Agent file | Model | Role description |
|------|-----------|-------|-----------------|
| **Queen** | _(orchestrator — no agent file; runs as the top-level Claude Code session)_ | opus | The orchestrator. Reads only briefings, verdict tables, commit messages, and agent notifications. Never reads source code or implementation templates. The only agent that pushes to remote. Spawns all subagents and makes all go/no-go decisions. |
| **Scout** | `agents/ant-farm-recon-planner.md` | opus | Pre-flight reconnaissance agent. Discovers ready and blocked tasks, builds a file modification matrix, assesses conflict risk, recommends specialist agent types, proposes 2–3 execution strategies with wave groupings, and writes the session briefing. |
| **Pantry** | `agents/ant-farm-prompt-composer.md` | opus | Prompt composition agent. Reads implementation templates, extracts per-task context from Scout metadata, writes data files and combined prompt previews, and returns a file-path table to the Queen. Keeps template content out of the Queen's context. |
| **Checkpoint Auditor** | `agents/ant-farm-checkpoint-auditor.md` | haiku (standalone: pre-spawn-check, scope-verify, session-complete), sonnet (team member: claims-vs-code, review-integrity) | Verification auditor. Runs all six checkpoints (startup-check, pre-spawn-check, scope-verify, claims-vs-code, review-integrity, session-complete) and writes timestamped audit reports to `{session-dir}/pc/`. Cross-checks orchestrator and agent work against ground truth. |
| **Crumb Gatherer** | _(spawned via `orchestration/templates/crumb-gatherer-skeleton.md` with a specialist `subagent_type`)_ | varies by task | Implementation agent. Executes exactly 6 mandatory steps: claim, design, implement, review, commit, summary doc. Constrained to its scope boundary. Documents adjacent issues without fixing them. |
| **Reviewer** | `agents/ant-farm-reviewer-{clarity,edge-cases,correctness,drift}.md` | opus (Correctness, Edge Cases), sonnet (Clarity, Drift) | Code review agent. Reads all changed files, catalogs findings with file:line references and severity, groups findings into preliminary root causes, and writes a structured review report. There are four Reviewer specializations: Clarity (P3), Edge Cases (P2), Correctness (P1–P2), and Drift (P3). The priority annotations (P1–P3) indicate the severity range each specialization focuses on — P1 is highest severity, P3 is lowest; see `orchestration/templates/reviews.md` for full definitions. Does not file issues — only Review Consolidator does. |
| **Review Consolidator** | `agents/ant-farm-review-consolidator.md` | opus | Consolidation reviewer. Reads all four Reviewer reports, merges duplicate findings, groups by root cause, documents merge rationale, files one issue per root cause with all affected surfaces, and writes a consolidated summary with deduplication log and priority breakdown. |
| **Spec Writer** | `agents/ant-farm-spec-writer.md` | opus | Requirements gathering agent. Reads freeform feature requests and existing codebase context, asks targeted clarifying questions via AskUserQuestion, and writes a structured spec.md with testable acceptance criteria. |
| **Researcher** | `agents/ant-farm-researcher.md` | sonnet | Parallel research specialist. Investigates a single focus area (Stack, Architecture, Pitfall, or Pattern) against a feature spec. Four Researcher instances run concurrently, each writing findings to a dedicated research file. Enforces a 100-line hard cap on output. |
| **Task Decomposer** | `agents/ant-farm-task-decomposer.md` | opus | Spec decomposition specialist. Reads spec.md plus research briefs and the existing codebase, then creates trails and crumbs via CLI, wires dependencies, and assigns scope. Produces a decomposition-brief.md summarizing every decision made. |
| **Session Scribe** | `agents/ant-farm-session-scribe.md` | sonnet | Session documentation agent. Reads all session artifacts — briefing, summaries, review reports, progress log, and git history — then writes the exec summary and CHANGELOG entry. The only agent that writes CHANGELOG.md. |

> **Historical Names** (pre-Wave 1–3 renames): Scout was `ant-farm-scout-organizer`, Pantry was `ant-farm-pantry-impl`, Checkpoint Auditor was `ant-farm-pest-control`, Reviewer was `ant-farm-nitpicker-{clarity,edge-cases,correctness,drift}`, Review Consolidator was `ant-farm-big-head`, Spec Writer was `ant-farm-surveyor`, Researcher was `ant-farm-forager`, Task Decomposer was `ant-farm-architect`, Session Scribe was `ant-farm-technical-writer`.
