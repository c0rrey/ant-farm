# Placeholder Conventions

This document defines the canonical placeholder conventions used across all orchestration templates and workflows.

## Overview

The orchestration system uses three distinct placeholder syntaxes, each with a specific purpose and substitution timing:

| Syntax | Purpose | Substituted By | Substituted When | Example |
|--------|---------|---|---|---------|
| `{UPPERCASE}` | Queen-provided context | Queen agent | Before agent spawn | `{TASK_ID}`, `{SESSION_DIR}`, `{EPIC_ID}` |
| `{lowercase-kebab}` | Runtime-derived or output format | Agent or system | At runtime by agent | `{session-dir}` (computed from SESSION_DIR), `{timestamp}` (from system clock) |
| `${SHELL_VAR}` | Bash/shell variables | Shell interpreter | When bash script executes | `${SESSION_ID}`, `${SESSION_DIR}` in code blocks only |

## Detailed Definitions

### Tier 1: Queen-Substituted (`{UPPERCASE}`)

These placeholders are filled in by the Queen agent **before spawning a subagent**. They represent information the Queen already possesses from prior steps.

**When to use**: Task context that came from a previous step (Scout metadata, bead data, etc.)

**Characteristics**:
- ALL CAPS with underscores
- Filled in by Queen in task skeleton templates
- Never appear in final prompts (all instances replaced with actual values)
- Used in data file paths, task IDs, epic IDs, file paths

**Examples**:
- `{TASK_ID}` — full bead ID including project prefix (e.g., `ant-farm-9oa`)
- `{TASK_SUFFIX}` — suffix portion only (e.g., `9oa` from `ant-farm-9oa`)
- `{EPIC_ID}` — epic suffix only (e.g., `74g`), or `_standalone` for tasks with no epic
- `{SESSION_DIR}` — session artifact directory path (e.g., `.beads/agent-summaries/_session-abc123`)
- `{AGENT_TYPE}` — subagent type for Task tool (e.g., `python-pro`, `general-purpose`)
- `{DATA_FILE_PATH}` — full path to data file pre-written by Pantry
- `{SUMMARY_OUTPUT_PATH}` — full path where agent should write summary doc
- `{EPIC_ID}` — epic suffix or `_standalone`

**Term Definition Block Template**:
Every template that uses `{UPPERCASE}` placeholders MUST include this block at the top:

```markdown
**Term definitions (canonical across all orchestration templates):**
- `{TASK_ID}` — full bead ID including project prefix (e.g., `ant-farm-9oa`)
- `{TASK_SUFFIX}` — suffix portion only, no project prefix (e.g., `9oa`)
- `{EPIC_ID}` — epic suffix only (e.g., `74g`), or `_standalone` for tasks with no epic
```

---

### Tier 2: Agent-Derived or Output Format (`{lowercase-kebab}`)

These placeholders represent values that are **computed at runtime by an agent** or are **output format specifications**. They appear in documentation, examples, and error messages—not in pre-spawn context.

**When to use**: Values computed during agent execution, output format examples, or placeholder text in instructions

**Characteristics**:
- lowercase with hyphens (kebab-case)
- Appear in agent-facing instructions or output specs
- Never pre-filled by Queen (agent computes or system provides them)
- Often represent file paths, IDs, or values derived from prior computations

**Examples**:
- `{session-dir}` — Derived from `{SESSION_DIR}` at runtime; used in agent-facing instructions to show "the session dir you were given" in relative terms
- `{task-id-suffix}` — Task suffix for output filenames
- `{timestamp}` — Generated by agent or system (e.g., `YYYYMMDD-HHMMSS`)
- `{N}` — Placeholder for a count or number in output specs (e.g., "Found {N} issues")
- `{type}` — Generic placeholder for review type in output filenames
- `{list}` — Generic placeholder for a list in documentation

---

### Tier 3: Shell Variables (`${SHELL_VAR}`)

These placeholders represent **bash/shell variables** and appear ONLY in code blocks where shell commands are executed.

**When to use**: In bash command blocks and shell scripts only

**Characteristics**:
- ALL CAPS with underscores
- Prefixed with `$` and wrapped in braces: `${VAR}`
- Only in code blocks (markdown ` ``` ` fences)
- Never in regular prose or placeholder text

**Examples**:
- `${SESSION_ID}` — Session ID value for bash commands
- `${SESSION_DIR}` — Session directory path for bash mkdir/mkdir operations
- `${EPIC_ID}` — Epic ID variable in bash scripts

**Usage pattern**:
```bash
SESSION_ID=$(date +%s | shasum | head -c 6)
SESSION_DIR=".beads/agent-summaries/_session-${SESSION_ID}"
mkdir -p ${SESSION_DIR}/{task-metadata,previews,prompts}
```

## Validation Rules

To ensure compliance with these conventions, apply these grep patterns:

### Pattern 1: Find all Tier 1 placeholders
```bash
grep -rE '\{[A-Z][A-Z_]*\}' orchestration/ --exclude-dir=_archive
```

Expected matches: `{TASK_ID}`, `{SESSION_DIR}`, `{EPIC_ID}`, etc. These should all appear in templates meant for Queen substitution or in term definition blocks.

### Pattern 2: Find all Tier 2 placeholders
```bash
grep -rE '\{[a-z][a-z\-]*\}' orchestration/ --exclude-dir=_archive
```

Expected matches: `{session-dir}`, `{timestamp}`, `{N}`, `{M}`, `{list}`, etc. These should appear in agent-facing instructions, examples, or output format specs.

### Pattern 3: Find all Tier 3 placeholders (shell variables)
```bash
grep -rE '\$\{[A-Z][A-Z_]*\}' orchestration/ --exclude-dir=_archive
```

Expected matches: `${SESSION_ID}`, `${SESSION_DIR}`. These should ONLY appear in bash code blocks (between ` ``` ` fences).

### Pattern 4: Detect invalid mixed casing
```bash
grep -rE '\{[A-Z]+[a-z_\-]+\}|\{[a-z\-]*[A-Z]+\}' orchestration/ --exclude-dir=_archive
```

This pattern finds mixed-case placeholders like `{MyVar}` or `{my_VAR}`, which should not exist. Any matches indicate violations.

### Why No Changes Needed

The development team unconsciously implemented the Tiered Placeholder Convention correctly:
- Queen-provided context uses uppercase (`{SESSION_DIR}`, `{TASK_ID}`)
- Agent-derived or output examples use lowercase (`{session-dir}`, `{timestamp}`)
- Shell variables use `${}` syntax (`${SESSION_ID}`)

The convention simply **documents** this existing best practice rather than imposing new restrictions.

## Benefits of This Convention

1. **Grep-able**: Each tier has a distinct regex pattern
2. **Self-documenting**: Casing alone tells you when substitution happens
3. **Reduces confusion**: New template authors don't wonder "will this be filled in?"
4. **Enables automation**: CI/CD can validate placeholder casing
5. **Flexible**: Accommodates current usage without massive refactor
6. **Semantically rich**: Three tiers map to three workflow phases

## Exceptions and Special Cases

### Shell variables in code blocks
Only location where Tier 3 (`${VAR}`) appears. These are NOT template placeholders — they are actual shell variables that will be expanded by the shell. They follow bash naming conventions (uppercase with underscores).

### Generic placeholders in examples
When showing an example output path like `.beads/agent-summaries/{path}`, the `{path}` is Tier 2 (agent-derived) even if it looks like a generic placeholder. If it's filled in by the Queen before spawn, it must be Tier 1 (`{PATH}` in uppercase).

### Legacy term definitions
Some files pre-date this convention. Their term definition blocks (if present) may use mixed casing. When updating these files, normalize to convention and regenerate term blocks.
